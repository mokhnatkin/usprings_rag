# CLAUDE.md - USprings RAG

Инструкции для работы над проектом. Дополняют глобальные правила пользователя.

## Что это

Пилотный RAG-проект: внутреннее хранилище инструкций + веб-портал с ИИ-ассистентом
для группы заводов УПЗ. Первый контур пользователей - ИТ-процессы, в частности 1С ERP.
Пользователь задаёт вопрос -> система ищет релевантные инструкции в векторной базе ->
LLM через OpenRouter формирует ответ с опорой на найденные документы.

Соответствует треку B6 ИТ-стратегии группы (см.
`C:\Users\Mokhnatkin\Documents\webdev\IT_strategy_usprings\IT_strategy\IT_strategy_2026-2029_ver0.4.md`).

## Принятые решения

- Стек: Python + FastAPI (бэкенд), PostgreSQL + pgvector (векторное хранилище),
  простой фронтенд (серверные шаблоны / HTML+JS; React - позже при необходимости),
  Docker Compose, on-premise. LLM - через OpenRouter.
- Белый список вопросов реализуется семантически: ответ выдаётся только при наличии
  релевантного документа выше порога сходства; иначе - вежливый отказ. Явного ручного
  списка тем нет.
- Основные форматы исходных инструкций - Word и PDF (часть форматов ещё не определена).
- Дополнительный источник инструкций - документация 1С:ИТС (its.1c.ru, есть подписка):
  парсинг работает, скрипты и гайд в `docs/1C_parsing/`, результаты - PDF с
  картинками (pandoc+typst). Скрипты параметризованы базой (`--db`), реестр -
  `registry-<db>.md`, PDF складываются сразу в папку коллекции.
  Учётные данные (`docs/1C_parsing/1C_portal_credentials.txt`) в `.gitignore`.
  Md-парсер для ingest - пост-MVP.
  - **1С:ERP** (`erp25ltsdoc`) - выгружен полностью (2026-07-14): реестр 3645 строк,
    411 документов, ~800 МБ, в `docs/manuals/its_erp/`; 2 главы недоступны на портале
    (битые ссылки, HTTP 404). Единица выгрузки - глава, строки реестра обычно якоря
    внутри неё.
  - **1С:ЗУП** (`zupcorpdoc`) - выгружен полностью (2026-07-14): реестр 194 строки,
    194 документа, ~12 МБ, в `docs/manuals/its_zup/`, ошибок нет. Оглавление плоское
    (строка = документ), дублей нет.
  - **Схема ссылок в оглавлении у баз разная**: у ERP разделы - `/db/<db>/bookmark/...`,
    у ЗУП - `/db/<db>/content/<N>/hdoc`. `build_registry.py` принимает обе формы.
    Пустой реестр у новой базы = снова другая схема, смотреть оглавление.
- Инструкции разных продуктов хранятся в **разных коллекциях**: коды `erp` и `zup`,
  в UI - «1С:ERP» и «1С:ЗУП». Пользователь выбирает коллекцию до вопроса, поиск идёт
  только по ней. **Справочник коллекций - в БД** (таблица `collections`: код,
  название, папка, порог, is_active; миграция 0004). `src/usprings_rag/collection.py` -
  тонкая read-model поверх неё (кэш в памяти), тот же объект для ingest, поиска, API
  и UI. Создание коллекции - `collections_service.create_collection` (строка + секция
  `chunks` + папка). Админ-UI справочника - этап 8 MVP1. Папка = коллекция
  (`docs/manuals/its_erp/`, `docs/manuals/its_zup/`). Ingest: `uv run ingest --collection erp`.
- Поиск изолирован на уровне БД: таблица `chunks` секционирована по коллекции
  (`PARTITION BY LIST`, миграция 0003), в каждой секции свой HNSW-индекс,
  `collection` денормализована в `chunks`. Иначе фильтр применяется уже после обхода
  индекса и top-k «съедают» соседние базы. Секции создаёт ingest, HNSW-индекс
  объявлен на родителе и наследуется секциями автоматически. PK `chunks` составной
  `(id, collection)`; `id` - последовательность (identity на секционированных
  таблицах есть только с PG17, у нас 16).
- Старый корпус `docs/manuals/IT_1C` (8 тестовых инструкций) удалён. Вместе с ним
  обнулены eval-набор и калибровка порога 0.53 - они были построены на нём;
  нужен новый golden-набор на корпусе ИТС (см. `docs/open-questions.md`).

## MVP1 (в работе)

Многопользовательский портал: аутентификация, роли, логирование, история, админка.
Направления и план - `docs/MVP/MVP1/` (`backlog.md`, `mvp-dev-plan.md`); статус по
этапам и что выяснилось - `docs/MVP/MVP1/mvp-dev-plan-progress.md` (источник истины).

Сделано (этапы 1-9, 2026-07-15):
- **Коллекции в БД** (таблица `collections`, `collection.py` - read-model).
- **Аутентификация**: внутренние учётки (логин+пароль, argon2), cookie-сессии
  (`SessionMiddleware`), bootstrap super-admin из `.env` (`SUPERADMIN_*`, `SECRET_KEY`).
  Портал закрыт за входом. Модули `security.py`, `auth.py`.
- **Роли и доступ**: `user` / `collection_admin` / `super_admin` (enum `Role`); доступ
  к коллекциям - таблица `user_collection_access`; `/collections` и `/ask*` учитывают
  доступ (403). Смена своего пароля - `/profile`.
- **Логирование** вопросов-ответов - таблица `query_log` (полный текст, токены,
  диагностика), best-effort (`logging_qa.py`). **Обратная связь** «ответ неверный» -
  `/feedback`. **История** - `/history`, `/api/history/*` (`history.py`).
- **Инструкции: загрузка и soft-delete** (этап 7). Soft-delete документов
  (`documents.archived_at`/`archived_by`, миграция 0008): архивный документ вне
  поиска (`retrieval.search` фильтрует `archived_at IS NULL`), возврат почти
  бесплатен (чанки на месте). **Фоновая индексация** загруженного PDF: таблица
  `index_job` + `indexing/jobs.py` (очередь) + `indexing/worker.py` (`IndexWorker` -
  поток в процессе приложения, одна задача за раз, переиспользует прогретую BGE-m3;
  зависшие `running` при старте -> `error`); флаг `INDEX_WORKER_ENABLED`. **Админ-экран
  документов** (collection_admin/super_admin): `admin/documents.py` + эндпоинты
  `/admin/documents`, `/api/admin/documents`, `/api/admin/jobs`,
  `/api/admin/documents/upload`, `.../archive|unarchive`; зависимость `require_admin`,
  per-collection - `check_collection_access(need_admin=True)`. Роль на клиенте -
  `GET /api/me`.

- **Справочники super-admin** (этап 8). **Пользователи** (`admin/users.py`):
  создание с ролью, автогрант `user` на активные коллекции, вкл/выкл, сброс пароля
  (временный), выдача доступа (`user_collection_access`); `require_super_admin`,
  защита от самодеактивации. **Коллекции** (`collections_service.update_collection`):
  правка `title`/`threshold`/`is_active` (code/folder неизменяемы), создание из UI
  через `create_collection`. **Журнал** (`admin/logs.py`, `require_admin`): просмотр
  `query_log` с фильтром коллекция/период и провалом в запись; super-admin - весь
  портал, collection-admin - свои коллекции. UI: `templates/admin/{users,collections,
  logs}.html`; меню по роли в шапке (`/api/me`).
- **Калибровка порогов из UI** (этап 9, super-admin). `admin/calibration.py`:
  прогон golden-набора (`EVAL_QUESTIONS_FILE`, дефолт `eval/questions.yaml`;
  в Docker-образ eval/ копируется) фоновой задачей со статусом (реестр в памяти,
  одна за раз), отчёт - recall, распределения, зазор, **рекомендованный порог по
  формализованному правилу MVP0**: `min(relevant) - 0.03, вниз до сотых`; негативы
  выше порога проходят сознательно (закрывает промпт) и фиксируются в обосновании.
  Живой прогон воспроизвёл калибровки 2026-07-14 (erp 0.58, zup 0.55). Экран
  `/admin/calibration`; применение порога - PATCH справочника коллекций (этап 8).

Дальше (не сделано): 10 - аналитика; 11 - навигация по ролям (финальная сверка);
12 - упаковка и документация. При выходе на них сверяться с планом и не
«додумывать» открытые решения молча.

## Открытые вопросы

Актуальный список - в `docs/open-questions.md`. Ключевые на сейчас:

- Приватность LLM: проверка соответствия политике безопасности перед продом
  (нужен zero-data-retention; на пилоте - платный тир OpenRouter).
- Полный перечень форматов исходных инструкций (пока подтверждены Word и PDF).
- Предел «белого списка»: околодоменный непокрытый вопрос вытягивает смежный чанк,
  и модель отвечает по нему. Смягчение (реранкер) - пост-MVP.

Порог сходства **больше не глобальный параметр `.env`**: он свойство коллекции
(поле `threshold` в таблице `collections`, читается через `collection.py`).
Действующие значения, откалиброванные на eval-наборах ИТС (2026-07-14): `erp` = 0.58,
`zup` = 0.55. Порог правится из UI super-admin (`/admin/collections`), калибровка с
рекомендованным порогом - `/admin/calibration` (этап 9; правило -
`min(relevant) - 0.03, вниз до сотых`). Прежние 0.53
(recall@1 = 17/17) получены на удалённом корпусе из 8 инструкций и недействительны.
Golden-наборы малы (16-18 вопросов) - пополнять реальными вопросами пилота.

на **платном** эндпоинте (выбрана по A/B; бесплатные `:free` часами отдают 429 и

План MVP0 (стек и этапы) - в `docs/MVP/MVP0/mvp-plan.md`. Ориентиры MVP0: локальные

Детали - в `docs/open-questions.md`.

## Как работать над проектом

- Инкрементально: маленькие проверяемые шаги, валидация каждого перед следующим.
- Python: пакеты через `uv` (`uv add`, `uv run`), не через pip/python напрямую.
- Не переусложнять, не программировать защитно. Сначала находить корневую причину,
  доказывать её, потом чинить.
- Актуализировать документацию (`docs/`, `README.md`) после изменений кода
  или когда узнаём что-то новое от пользователя.
- UI и тексты для пользователя - на русском (клиенты в Казахстане/РФ).
- Без эмодзи в коде, логах и выводе.
