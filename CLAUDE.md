# CLAUDE.md - USprings RAG

Инструкции для работы над проектом. Дополняют глобальные правила пользователя.

## Что это

Пилотный RAG-проект: внутреннее хранилище инструкций + веб-портал с ИИ-ассистентом
для группы заводов УПЗ. Первый контур пользователей - ИТ-процессы, в частности 1С ERP.
Пользователь задаёт вопрос -> система ищет релевантные инструкции в векторной базе ->
LLM через OpenRouter формирует ответ с опорой на найденные документы.

Соответствует треку B6 ИТ-стратегии группы (см.
`C:\Users\Mokhnatkin\Documents\webdev\IT_strategy_usprings\IT_strategy\IT_strategy_2026-2029_ver0.4.md`).

## Принятые решения

- Стек: Python + FastAPI (бэкенд), PostgreSQL + pgvector (векторное хранилище),
  простой фронтенд (серверные шаблоны / HTML+JS; React - позже при необходимости),
  Docker Compose, on-premise. LLM - через OpenRouter.
- Белый список вопросов реализуется семантически: ответ выдаётся только при наличии
  релевантного документа выше порога сходства; иначе - вежливый отказ. Явного ручного
  списка тем нет.
- Основные форматы исходных инструкций - Word и PDF (часть форматов ещё не определена).

## Открытые вопросы

Актуальный список - в `docs/open-questions.md`. Ключевые на сейчас:

- Приватность LLM: проверка соответствия политике безопасности перед продом
  (нужен zero-data-retention; на пилоте - платный тир OpenRouter).
- Полный перечень форматов исходных инструкций (пока подтверждены Word и PDF).
- Предел «белого списка»: околодоменный непокрытый вопрос вытягивает смежный чанк,
  и модель отвечает по нему. Смягчение (реранкер) - пост-MVP.

Уже решено (не пересматривать без причины): порог `SIMILARITY_THRESHOLD=0.53`
(откалиброван на eval, recall@1 = 17/17); LLM - `qwen/qwen3-next-80b-a3b-instruct`
на **платном** эндпоинте (выбрана по A/B; бесплатные `:free` часами отдают 429 и
для работы непригодны).

План MVP (стек и этапы) - в `docs/MVP/MVP0/mvp-plan.md`. Ориентиры MVP уже приняты:
локальные эмбеддинги BGE-m3, индексируем только текст (без OCR), загрузка через CLI-скрипт,
без авторизации; детали - в `docs/open-questions.md`.

Не «додумывай» открытые решения молча - при выходе на них уточняй у пользователя.

## Как работать над проектом

- Инкрементально: маленькие проверяемые шаги, валидация каждого перед следующим.
- Python: пакеты через `uv` (`uv add`, `uv run`), не через pip/python напрямую.
- Не переусложнять, не программировать защитно. Сначала находить корневую причину,
  доказывать её, потом чинить.
- Актуализировать документацию (`docs/`, `README.md`) после изменений кода
  или когда узнаём что-то новое от пользователя.
- UI и тексты для пользователя - на русском (клиенты в Казахстане/РФ).
- Без эмодзи в коде, логах и выводе.
