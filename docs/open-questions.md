# Открытые вопросы и решения

Живой список. Решённое перемещаем в раздел «Принятые решения» с датой.

## Открытые

- **A/B-сравнение LLM (подшаг 6.4) - не выполнено, заблокировано лимитом.** 2026-07-12
  обе плановые модели (`qwen/qwen3-next-80b-a3b-instruct:free` и
  `google/gemma-4-31b-it:free`) стабильно отдавали `429 rate-limited upstream` в течение
  всей сессии - у разных провайдеров (Venice, Google AI Studio, OpenInference). Лимит
  не наш: ключ не free-tier, кредиты не израсходованы (`usage: 0`), а соседняя
  бесплатная модель `google/gemma-4-26b-a4b-it:free` с тем же ключом отвечала штатно
  (на ней и проверен сквозной сценарий этапа 6). Повторить A/B, когда бесплатные
  эндпоинты освободятся; модель по умолчанию (`OPENROUTER_MODEL`) до этого **не меняем**
  без решения владельца. Практический вывод для пилота: доступность бесплатного тира
  нестабильна - это аргумент к платному тиру на проде (см. пункт про приватность ниже).
- **Источники при ответе «информации нет».** Если околодоменный вопрос прошёл порог,
  но LLM честно ответила, что информации нет, портал всё равно показывает источники
  найденных чанков (пример: «инвентаризация склада» -> «Планирование отгрузки, стр.6-7»).
  Выглядит нелогично. Решить на этапе 7 (портал): либо не показывать источники в таком
  ответе, либо просить LLM явно помечать «нет ответа» отдельным признаком.
- **Форматы исходных инструкций.** Подтверждены Word и PDF; полный перечень не собран.
  Собрать остальные образцы, когда появятся.
- **Порог на реальных вопросах пилота.** `SIMILARITY_THRESHOLD=0.53` откалиброван на
  eval-наборе из 25 вопросов (см. принятые решения от 2026-07-12); лучшее сходство
  каждого запроса пишется в лог с этапа 5 - по мере накопления реальных вопросов
  пользователей распределение перепроверить. Если понадобится жёстче резать
  околодоменные непокрытые вопросы - кандидат на пост-MVP реранкер (cross-encoder).
  Регламент перекалибровки (2026-07-12): по триггерам, не по календарю - новые
  инструкции того же домена требуют только eval-прогона (recall@1 + хвост
  распределения), обязательная перекалибровка - при новой коллекции, смене модели
  эмбеддингов или чанкинга; обзор логов - после ~100 реальных вопросов, далее раз
  в месяц-два. Подробная процедура - в `maintenance.md`.
- **Хостинг LLM.** OpenRouter (облачные модели) - зафиксировано для пилота. Соответствие
  политике безопасности (какие данные уходят в облако) - проверить перед продом.
- **Приватность бесплатного тира OpenRouter.** У бесплатных моделей провайдер, как правило,
  может логировать/обучаться на запросах, а в промпт уходят внутренние инструкции завода.
  Для калибровки на образцах - приемлемо; для прода нужен платный тир с zero-data-retention
  или выбором провайдера без обучения на данных. Проверить перед продом.
- **Модель разграничения доступа** (внутренние учётки, LDAP/AD, роли) - детально
  проработать на пост-MVP. Общее решение по видам авторизации принято (см. ниже).
- **Модель разделения баз знаний по подразделениям** - детали при подключении других
  служб. Технический задел (поле `collection`) закладываем на MVP (см. ниже).

## Принятые решения

- **2026-07-12. `SIMILARITY_THRESHOLD=0.53` (калибровка этапа 5).** Eval: 25 вопросов
  (11 покрытых, 6 перефразированных, 8 нерелевантных), recall@1 = 17/17 - все
  релевантные вопросы приводят верный документ на первое место. Распределения лучших
  сходств: релевантные 0.548-0.813, нерелевантные 0.288-0.561. Ключевой факт:
  **чистого разделения нет** - околодоменные непокрытые 1С-вопросы («инвентаризация
  склада в 1С» 0.547, «права доступа в 1С» 0.561) ложатся внутрь релевантной зоны,
  потому что корпус реально содержит близкую лексику (роли, склад, остатки); вопрос
  «покрыто ли это нашими 8 инструкциями» из семантики вопроса не извлекается.
  Решение - порог 0.53 защищает релевантный хвост (min 0.548, запас 0.018) и
  отсекает все вне-доменные вопросы (максимум - «резервная копия базы 1С» 0.521;
  погода/кухня/HR/VPN - 0.29-0.48). Асимметрия ошибок осознанная: ложный отказ
  ничем не компенсируется и бьёт по доверию к пилоту, а ложный пропуск
  околодоменного вопроса страхуется промптом этапа 6 («если ответа во фрагментах
  нет - честно сказать»). Следствие: «белый список» по порогу отвечает за отсечение
  вне-доменного, честность на околодоменном непокрытом - зона ответственности
  промпта. Приёмку этапа 6 («отказ без вызова LLM») проверять на вне-доменных
  вопросах; околодоменные дадут LLM-ответ «информации нет» - это ожидаемое
  поведение, не дефект. Прежний дефолт 0.5 пропускал и «резервную копию» (0.521);
  0.53 доминирует его по всем измерениям. Порог выше 0.548 недопустим - режет
  живой вопрос про частичное выполнение операции.
- **2026-07-07. Стек пилота.** FastAPI (бэкенд), PostgreSQL + pgvector (векторное хранилище),
  простой фронтенд (серверные шаблоны / HTML+JS, React - позже), Docker Compose, on-premise;
  LLM через OpenRouter.
- **2026-07-07. Белый список вопросов.** Реализуется семантически по порогу сходства,
  без ручного перечня тем.
- **2026-07-10. Эмбеддинги на MVP.** Локальная multilingual-модель BGE-m3 (on-premise,
  данные не уходят наружу). Провайдера эмбеддингов абстрагируем, чтобы позже сменить
  на API без переработки пайплайна. Итоговый выбор можно пересмотреть перед продом.
- **2026-07-10. Скриншоты и OCR на MVP.** Индексируем только текстовый слой PDF; OCR
  и мультимодальная обработка скриншотов не нужны (у образцов текст извлекается, действия
  продублированы прозой). В ответе даём ссылку на исходный PDF. OCR/мультимодальность - пост-MVP.
- **2026-07-10. Загрузка инструкций на MVP.** Наполнение базы CLI-скриптом ingest по PDF
  из папки. Админ-UI загрузки - пост-MVP.
- **2026-07-10. Авторизация.** На MVP не требуется (открытый доступ). Пост-MVP - два вида:
  внутренняя и внешняя через LDAP / Microsoft AD; архитектуру не ломать под добавление аутентификации.
- **2026-07-10. Задел под масштабирование.** В схеме `documents` заранее поле `collection`
  (значение по умолчанию `it_1c`), чтобы позже разделять базы знаний по подразделениям без миграции.
- **2026-07-10. LLM-модель по умолчанию на MVP.** `qwen/qwen3-next-80b-a3b-instruct:free`
  (бесплатная на OpenRouter): сильный русский, точное следование инструкции «отвечай только
  по контексту», MoE (быстрый инференс), контекст 262K. Альтернатива для A/B на калибровке -
  `google/gemma-4-31b-it:free`. ID модели выносим в `.env` (`OPENROUTER_MODEL`), смена без
  переработки кода. Итоговый выбор пересматриваем на калибровке и перед продом (см. открытые:
  приватность бесплатного тира, лимиты free ~20 req/min).
- **2026-07-10. Бесплатные LLM на разработке и MVP.** На всём протяжении разработки и на
  пилоте используем только бесплатные модели OpenRouter. Переход на платный тир - отдельное
  решение для прода (приватность/лимиты, см. открытые вопросы выше).
- **2026-07-11. Калибровка порога - в составе этапа 5, до генерации.** Eval-набор
  вопросов, recall@k, распределения сходств и выбор `SIMILARITY_THRESHOLD` делаем
  сразу после семантического поиска: приёмка этапа 6 («отказ на нерелевантных»)
  зависит от порога, калибровать его после генерации поздно. A/B по LLM - в составе
  этапа 6.
- **2026-07-11. `source_path` - относительный путь.** Храним путь к PDF относительно
  корня папки инструкций (POSIX-разделители), чтобы он совпадал на хосте и в
  Docker-контейнере и годился для раздачи файла из API. Нормализация текущих
  данных - на этапе 7 (правка pipeline + повторный ingest c предварительной
  очисткой старых записей `TRUNCATE documents CASCADE`: идемпотентность ключуется
  по `source_path`, после смены путей старые записи дали бы дубли).
- **2026-07-11. Парсинг Word (docx) - пост-MVP.** Word подтверждён как основной
  формат наряду с PDF (см. `overview.md`), но все образцы пилота - PDF. На MVP
  индексируем только PDF; docx добавляем пост-MVP («парсер по расширению»).
- **2026-07-10. ORM и миграции - SQLAlchemy + Alembic с самого начала.** Хотя на MVP таблиц
  мало, проект пост-MVP разрастётся (логирование, аналитика, несколько баз знаний). Единая
  ORM-модель и миграции с первого этапа окупаются и избавляют от переписывания. Тип `vector`
  подключаем через пакет `pgvector` (SQLAlchemy-тип + адаптер).
