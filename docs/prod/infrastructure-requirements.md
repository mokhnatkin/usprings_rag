# Требования к инфраструктуре (оценка)

Ориентировочная оценка ресурсов для staging и prod на MVP и пост-MVP. Сделана «по
архитектуре», не по замерам - цифры уточняются на пилоте (см. оговорки в конце).
Дополняет `docs/overview.md` и `docs/MVP/MVP0/mvp-dev-plan.md`.

## Что определяет ресурсы

Главный потребитель - не LLM, а локальная модель эмбеддингов **BGE-m3**. LLM
работает через OpenRouter (внешний вызов), поэтому **GPU для генерации ответа не
нужен**. Остальное (FastAPI, Postgres+pgvector на пилотном объёме) - лёгкое.

| Компонент | Нагрузка | Комментарий |
|-|-|-|
| BGE-m3 (эмбеддинги, in-process) | CPU + ~3-4 ГБ RAM | Веса ~2.3 ГБ + оверхед torch. На каждый вопрос - 1 инференс на CPU (~0.1-0.5 с). Главный фактор и узкий ресурс при росте нагрузки. |
| PostgreSQL + pgvector (HNSW) | RAM под индекс | На пилоте ничтожно. Растёт линейно с числом чанков: ~1 млн векторов ≈ 4 ГБ сырых данных + оверхед графа HNSW (~1.5x). |
| FastAPI + оркестрация | Незначительно | I/O-bound, основное время - ожидание ответа OpenRouter. |
| LLM (OpenRouter) | 0 локально | Внешний вызов. GPU не требуется. |

## Оценки (on-premise, Docker Compose)

### MVP prod
Одна база `it_1c`, без авторизации, единичные пользователи.

- vCPU: **4** (2 - впритык; 4 - комфортно для эмбеддингов).
- RAM: **8 ГБ** (BGE-m3 ~4 ГБ + Postgres + app + ОС; 4 ГБ рискованно).
- Диск: **30 ГБ SSD** (веса модели + образы docker с torch + БД + PDF).
- GPU: **не нужен**. Всё на одной VM.

### MVP staging
Тот же класс, можно урезать: **2-4 vCPU / 6-8 ГБ / 20 ГБ**. Держать сопоставимым с
prod, иначе тесты производительности не показательны.

### Post-MVP prod
Авторизация LDAP/AD, логирование + аналитика, несколько баз знаний по подразделениям,
умеренная одновременная нагрузка ~10-30 пользователей, корпус до ~100-500 тыс. чанков.

- vCPU: **8** (пропускная способность эмбеддингов на CPU + аналитические запросы).
- RAM: **16 ГБ** (эмбеддинги ~4 ГБ + растущий HNSW-индекс + логи/аналитика + запас).
- Диск: **50-100 ГБ SSD** (рост БД за счёт логов запросов, больше документов, бэкапы).
- GPU: **опционально**, оправдан только при высокой одновременной нагрузке или
  добавлении реранкера.

### Post-MVP staging
Уменьшенная копия prod: **4 vCPU / 8-12 ГБ / 40 ГБ**.

## Драйверы масштабирования (когда потребуется больше)

- **Одновременная нагрузка.** Эмбеддинги на CPU - последовательный узкий ресурс.
  Десятки одновременных запросов -> добавлять ядра или вынести эмбеддинги в отдельный
  сервис. Сотни одновременных или требование низкой задержки -> единичный GPU
  (T4/L4 16 ГБ) под эмбеддинги/реранкер.
- **Размер корпуса.** Пока это внутренние инструкции - вероятно, десятки-сотни тысяч
  чанков, pgvector с HNSW в RAM тянет спокойно. Миллионы векторов -> планировать RAM
  под индекс отдельно.
- **Аналитика.** Со временем - вынести БД логов/аналитики на отдельный том или
  read-replica, чтобы аналитические запросы не мешали основному потоку.

## Оговорки

Оценка по архитектуре, не замер. Две главные неизвестные, от которых зависят цифры:

1. **Пиковая одновременная нагрузка** - сколько пользователей реально спрашивают
   одновременно.
2. **Итоговый объём корпуса** пост-MVP - сколько баз знаний и документов.

Обе уточняются на пилоте и при подключении новых подразделений. На калибровке
(этап 6, см. `mvp-dev-plan.md`) заодно измерим фактическую задержку и память BGE-m3
на целевом железе - тогда цифры можно ужесточить.
