# Как работает база знаний (механика RAG)

Пояснение для команды разработки: как устроена система после развёртывания,
как пополняется база знаний, что и когда векторизуется, как отвечает ассистент.
Архитектура и требования - в `overview.md`, планы реализации - в `MVP/MVP0/`.

## Общий принцип

Система построена по схеме RAG (Retrieval-Augmented Generation): знания живут
в базе данных, а не в весах моделей. Ответ на вопрос собирается в два шага -
сначала **поиск** релевантных фрагментов инструкций в векторной базе, затем
**генерация** ответа LLM строго по найденным фрагментам.

Ключевое следствие: **никакие модели ничему не обучаются**. «Система узнала
новое» = «в базу добавили документ»; «забыла» = «документ удалили». Актуализация
знаний - операция за минуты, без разметки, переобучения и GPU.

## Два независимых контура

### Контур 1. Наполнение базы (ingest, редкий, офлайн)

```
PDF -> извлечение текста (pypdf) -> чанкинг (смысловые секции)
    -> векторизация чанков (BGE-m3, локально) -> запись в PostgreSQL/pgvector
```

- Запускается администратором командой (`uv run ingest` / `docker compose run app ingest`).
- Идемпотентен по хэшу файла: прогон можно запускать по всей папке хоть каждый
  день - неизменённые файлы пропускаются за миллисекунды, обрабатываются только
  новые и изменённые. Портал при этом не останавливается: новые чанки появляются
  в БД, поиск сразу их видит.
- Каждый документ разбивается на чанки (фрагменты в сотни токенов) с привязкой
  к страницам - на этом строятся ссылки на источник в ответе.

### Контур 2. Ответ на вопрос (онлайн, на каждый запрос)

```
вопрос -> векторизация вопроса (BGE-m3, та же модель)
       -> top-k ближайших чанков в pgvector (косинусное сходство)
       -> порог сходства: ниже - вежливый отказ (LLM не вызывается)
       -> выше - промпт с найденными чанками -> LLM (OpenRouter) -> ответ + ссылки на PDF
```

- «Белый список вопросов» реализован порогом сходства, без ручного перечня тем:
  нет релевантного документа - нет ответа. Отказ мгновенный (доли секунды),
  токены LLM не тратятся, галлюцинации на нерелевантном исключены.
- Источники в ответе формируются из метаданных чанков (документ, страницы),
  а не генерируются LLM - придумать несуществующую ссылку модель не может.

## Пополнение базы знаний

**MVP:** администратор копирует PDF в папку инструкций на сервере (смонтирована
в контейнер приложения) и запускает ingest одной командой. «Добавить 5 новых
инструкций» = скопировать 5 файлов + одна команда.

**Пост-MVP:** админский веб-модуль - загрузка файла через браузер, сервер сам
вызывает тот же ingest-пайплайн. Меняется только «кто нажимает кнопку», код
пайплайна общий. Плюс управление: список документов, удаление, разделение
баз знаний по подразделениям (поле `collection` уже заложено в схеме).

**Масштаб:** сотни инструкций - это тысячи чанков. Для pgvector с HNSW-индексом
это крошечный объём (индекс рассчитан на миллионы векторов); векторизация сотни
документов на CPU - минуты, и выполняется один раз на документ, а не на каждый
вопрос.

**Нужна ли перекалибровка порога при пополнении?** Как правило, нет: порог
разделяет распределения сходств, которые зависят от модели эмбеддингов и стиля
документов, а не от их количества. При добавлении инструкций того же домена
достаточно дописать пару вопросов в eval-набор и прогнать `run_eval.py`
(~минута): recall@1 не упал, релевантный хвост с запасом над порогом - порог
не трогаем. Обязательная перекалибровка - при новой предметной области
(коллекции), смене модели эмбеддингов или параметров чанкинга. Подробные
регламенты и триггеры - в `maintenance.md`.

## Векторизация: какая модель и почему одна

Всю векторизацию делает **одна и та же локальная модель BGE-m3** (multilingual,
dim 1024), развёрнутая на сервере: и чанки при ingest, и каждый вопрос
пользователя. Это жёсткое требование: вопрос и чанки должны быть в одном
векторном пространстве, иначе поиск сломается.

Следствия:

- Данные при векторизации **не покидают контур компании** (модель локальная).
- Смена модели эмбеддингов = перевекторизация всей базы. Для этого провайдер
  абстрагирован интерфейсом `EmbeddingProvider`; при наших объёмах пересборка -
  минуты-часы, не проблема.
- BGE-m3 не дообучается и не настраивается - работает «из коробки».

LLM (через OpenRouter) тоже не обучается на наших данных в рамках работы
системы: она получает найденные фрагменты в промпте каждого запроса и отвечает
по ним. (Отдельный вопрос - политика логирования бесплатного тира OpenRouter,
см. `open-questions.md`: для прода - платный тир с zero-data-retention.)

## ИИ-ассистент: интерфейс и скорость

**Интерфейс.** Чат-окно «вопрос-ответ». На MVP каждый вопрос обрабатывается
независимо, без учёта предыдущих реплик (история видна на экране, но в поиск
не попадает). Настоящий многоходовый диалог требует переформулировки вопроса
с учётом истории (query rewriting) - это пост-MVP; ценность проверяем на пилоте
по логам: для базы инструкций большинство обращений - одиночные «как сделать X».

**Скорость.** Составляющие задержки ответа:

| Шаг | Время | Комментарий |
|-|-|-|
| Векторизация вопроса (BGE-m3, CPU) | ~0,1-0,5 с | при прогретой модели |
| Поиск в pgvector | миллисекунды | на наших объёмах незаметен |
| Генерация LLM | 3-15 с | доминирует над всем остальным |

Что из этого следует:

- **Стриминг (SSE)** - реализован (`GET /ask/stream`): текст показывается по мере
  генерации, не надо ждать ответ целиком. На бесплатной модели первые слова приходят
  через ~5 с (провайдер сам долго отдаёт первый токен); на быстрой модели эффект будет
  заметнее. Отказ по порогу остаётся мгновенным - приходит сразу, без вызова LLM.
- **Прогрев модели эмбеддингов при старте приложения** (lifespan-хук FastAPI) -
  иначе первый запрос платит десятки секунд за загрузку весов.
- **Отказы мгновенны** - LLM для них не вызывается.
- На бесплатном тире OpenRouter лимит ~20 запросов/мин и возможны очереди у
  провайдера - для пилота приемлемо; при росте числа пользователей платный тир
  нужен и для приватности, и для стабильной скорости.
