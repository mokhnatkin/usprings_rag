# Регламенты развития и сопровождения портала

Живой документ: подробные регламенты типовых операций по мере роста базы знаний
и портала. Новые регламенты (авторизация, Word-инструкции, админ-модуль и пр.)
дописываем сюда по мере появления. Механика системы - в `how-it-works.md`,
архитектура - в `overview.md`.

Сводка триггеров - что делать при каком событии:

| Событие | Ingest | Eval-прогон | Калибровка порога |
|-|-|-|-|
| Новые инструкции того же домена | да | да (дёшево, ~минута) | нет, если eval чист |
| Обновление/удаление инструкции | да | по желанию | нет |
| Новый процесс / подразделение (коллекция) | да | да, свой набор вопросов | **обязательно** |
| Смена модели эмбеддингов | полная пересборка базы | да | **обязательно, с нуля** |
| Смена параметров чанкинга | полная пересборка базы | да | **обязательно** |
| Смена LLM (A/B, раздел 5) | нет | нет | нет - поиск не затрагивается |
| Сигналы из логов эксплуатации | - | да | по результатам |

## 1. Добавление новых инструкций (в существующую коллекцию)

Типовой случай: новые PDF в существующую базу знаний (`erp` или `zup`).

Откуда берутся инструкции: собственные документы (Word/PDF) и документация
1С:ИТС - её выгружает парсер сразу в папку нужной коллекции
(`uv run --with requests --with beautifulsoup4 python docs/1C_parsing/its_parse_all.py --db zupcorpdoc`,
см. `docs/1C_parsing/parsing-guide.md`) в двух вариантах: markdown (только
текст) или PDF с картинками (текст + скриншоты, сборка pandoc+typst). Ingest
принимает только PDF, поэтому PDF-вариант индексируется сразу. Markdown - после
добавления md-парсера («парсер по расширению», как запланировано для docx).

1. Скопировать файлы в папку коллекции (`docs/manuals/its_erp/`,
   `docs/manuals/its_zup/`; папка = коллекция).
2. Прогнать ingest для этой коллекции: `uv run ingest --collection erp`
   (`docker compose run --rm app ingest --collection erp`). Папка берётся из
   справочника коллекций, указывать её не нужно. Идемпотентен: неизменённые
   файлы пропускаются, обрабатываются только новые и изменённые. Портал не
   останавливается.
3. Дописать в `eval/questions.yaml` по 1-2 вопроса на новый документ:
   один прямой (формулировка близка к тексту, с `expected_doc`) и желательно
   один разговорный (как спросил бы пользователь, без терминов документа).
4. Прогнать eval по этой коллекции: `uv run --no-sync python eval/run_eval.py`. Смотреть:
   - **recall@1 не упал** - верные документы по-прежнему первые. Это главная
     метрика, деградирующая с ростом корпуса (больше конкурирующих чанков);
   - **релевантный хвост** - min(relevant) остаётся с запасом над порогом коллекции.
5. Если обе проверки чисты - готово, порог не трогаем. Если нет - раздел 4.

Порог устойчив к росту корпуса: он разделяет распределения сходств, которые
зависят от модели эмбеддингов и стиля документов, а не от их количества.

Конкретные цифры хвоста (прежние 0.53 / 0.548) относились к удалённому тестовому
корпусу и больше не действуют - ориентиры появятся после калибровки на корпусе ИТС.

## 2. Обновление и удаление инструкций

- **Обновление**: заменить файл в папке (то же имя) и прогнать ingest - изменение
  ловится по хэшу, старые чанки удаляются каскадом, документ векторизуется заново.
- **Переименование/перемещение файла**: идемпотентность ключуется по `source_path` -
  старая запись останется рядом с новой. Удалить старую вручную
  (`DELETE FROM documents WHERE source_path = '...'`; каскад уберёт чанки)
  или пересобрать коллекцию целиком
  (`DELETE FROM documents WHERE collection = 'erp'` + ingest этой коллекции).
- **Пометка неактуальным (архивация)**: из UI (`/admin/documents`) администратор
  коллекции архивирует документ - он исключается из поиска (фильтр
  `archived_at IS NULL`), но остаётся в БД для аудита и старых логов; возврат из
  архива - кнопкой (переиндексация не нужна, чанки на месте). Это штатный способ
  «убрать» инструкцию (MVP1, этап 7).
- **Полное удаление** из UI не предусмотрено (предпочтителен soft-delete). Если
  всё же нужно физически убрать документ - вручную из БД
  (`DELETE FROM documents WHERE source_path = '...'`; каскад уберёт чанки) и удалить
  файл из папки коллекции.

## 3. Подключение нового процесса / подразделения (новая коллекция)

Новая предметная область (HR, бухгалтерия, другой завод) - это новая коллекция,
а не просто новые файлы. Коллекция - единица изоляции поиска: чанки
секционированы по ней, и запрос не выходит за границы выбранной базы.

1. Создать коллекцию из UI super-admin: `/admin/collections` -> «Новая коллекция»
   (код, название, папка, стартовый порог). Сервис `create_collection` атомарно
   заводит строку справочника, секцию `chunks_<код>` (`PARTITION OF`, HNSW-индекс
   Postgres наследует от родителя) и папку инструкций - ручная DDL и правка кода
   не нужны. Справочник живёт в таблице `collections`; `collection.py` - read-model
   поверх неё (кэш сбрасывается при правке из UI).
2. Наполнить коллекцию инструкциями (PDF):
   - по одному файлу - из UI (`/admin/documents` -> загрузка): файл кладётся в папку
     коллекции, индексируется фоновой задачей со статусом;
   - массово - CLI: положить PDF в `docs/manuals/<папка>/` и прогнать
     `uv run ingest --collection <код>` (идемпотентно по `source_path`).
3. **Обязательная калибровка для новой коллекции** (раздел 4): другой стиль и
   лексика документов дают другие распределения сходств; чужой порог для них
   не проверен. Свой блок вопросов в eval (covered / paraphrased / irrelevant,
   включая околодоменные негативы новой области).

Справочник коллекций и разграничение доступа к ним ведутся в БД (MVP1): коллекция -
единица изоляции поиска и единица доступа (таблица `user_collection_access`).

## 4. Калибровка порога (процедура)

**Порог - свойство коллекции**, а не одно глобальное число: он хранится в таблице
`collections` (поле `threshold`, не в `.env` и не в коде) и калибруется отдельно на
eval-наборе своей коллекции. Правит его super-admin из UI (`/admin/collections`) -
изменение действует в поиске сразу, без деплоя (read-model сбрасывает кэш). Полный
разбор первой калибровки (на удалённом тестовом корпусе, как пример процедуры):
`open-questions.md`, решение от 2026-07-12.

Штатный путь - **экран калибровки** (super-admin, `/admin/calibration`, этап 9
MVP1): выбор коллекции -> «Запустить калибровку» -> фоновый прогон golden-набора ->
отчёт (recall, распределения, зазор, **рекомендованный порог с обоснованием**) ->
«Применить рекомендованный» или свой. Рекомендация считается по формализованному
правилу MVP0: `порог = min(relevant) - 0.03, округлённый вниз до сотых`; негативы
выше порога рекомендацию не двигают (околодоменные проходят сознательно - их
закрывает промпт), recall@1 < 100% по covered даёт предупреждение «сначала чинить
поиск». CLI-прогон `eval/run_eval.py` остаётся для разработки и отладки.

1. Актуализировать `eval/questions.yaml` (путь настраивается ключом
   `EVAL_QUESTIONS_FILE`). Состав: покрытые вопросы с ожидаемым документом,
   разговорные перефразировки, нерелевантные. Обязательно включать
   **трудные негативы** - непокрытые вопросы рядом с доменом (про 1С, завод,
   смежные процессы): именно они, а не «погода», определяют нижнюю границу порога.
2. Запустить калибровку из UI (`/admin/calibration`) - или, при разработке,
   `uv run --no-sync python eval/run_eval.py`.
3. Проверить рекомендацию по критериям (они же зашиты в обоснование отчёта):
   - recall@1 близок к 100% - иначе сначала чинить поиск (чанкинг, формулировки),
     порог не спасёт;
   - порог ниже min(relevant) с запасом - **ложный отказ хуже ложного пропуска**:
     отказ на легитимный вопрос ничем не компенсируется и бьёт по доверию,
     а просочившийся околодоменный вопрос страхуется промптом
     («если ответа во фрагментах нет - честно сказать»);
   - порог выше максимума вне-доменных негативов;
   - околодоменные негативы могут проходить порог - это принятое поведение,
     их закрывает промпт, не порог.
   Смотреть распределения целиком, не только рекомендованное число: на малом
   наборе зазор может быть совпадением (у erp - 0.006).
4. Зафиксировать: применить порог кнопкой на экране калибровки (или в
   `/admin/collections`) - он ляжет в `collections.threshold` и подействует сразу;
   решение с данными - в `open-questions.md`, факт - в журнале прогресса.

**Golden-наборы малы - пополнять реальными вопросами пилота.** На 2026-07-15 в
`eval/questions.yaml` по 16 (erp) и 18 (zup) вопросов. Пороги на них откалиброваны,
но этого объёма мало: у erp зазор между релевантными и негативами вышел 0.006 - на
такой выборке это совпадение, а не устойчивая граница. Как только пойдут реальные
вопросы пилота, добавлять их в набор (раздел 7 - какие именно ловить в логах;
подтверждённые жалобы - раздел «Разбор жалоб»). Настоящая уверенность в порогах
появится только на живом трафике, синтетический набор - лишь стартовая калибровка.

## 5. Смена LLM (A/B-сравнение моделей)

Занимает **минуты**, если следовать регламенту. Первый A/B (2026-07-12) занял два часа,
из которых полезной работы было ~2 минуты - ниже разобрано, что именно тратило время.

```bash
# 1. Кэш контекстов - один раз (пересобрать после ingest / смены порога / questions.yaml)
uv run --no-sync python eval/build_contexts.py

# 2. Сравнение - секунды, вызовы идут параллельно
uv run --no-sync python eval/run_ab.py <модель-A> <модель-B> [<модель-C> ...]
```

Скрипт печатает таблицу со счётом и список замечаний, полный отчёт с ответами -
в `eval/ab-report.md`. Смотреть в первую очередь на счёт и замечания, ответы читать
только по спорным вопросам.

**Правила, выведенные из первого A/B:**

- **Только платные эндпоинты.** Бесплатные (`:free`) часами отдают `429 rate-limited
  upstream` - на них нельзя измерять. Полный A/B стоит **доли цента** (первый обошёлся
  в $0.0044): экономия бессмысленна, а ожидание съедает часы.
- **Поиск считается один раз** (`build_contexts.py`). A/B сравнивает LLM, а не поиск;
  гонять BGE-m3 (2,3 ГБ, ~минута загрузки) под каждую модель незачем. Побочные плюсы:
  `run_ab.py` не тянет torch (нет висящих процессов - хроническая беда проекта) и
  контекст у моделей гарантированно идентичен.
- **Проверяем автоматически, а не глазами.** Ручное чтение медленно и **пропускает
  дефекты**: при чтении глазами не был замечен искажённый маркер (`НCЕТ_ОТВЕТА` с
  латинской `C`), из-за которого служебное слово ушло бы пользователю. Автопроверки
  нашли это за секунды. Что проверяется: маркер `NO_ANSWER` там, где ответа в контексте
  нет (и его отсутствие там, где есть), `must_contain` из `questions.yaml`, отсутствие
  markdown, неодносложность.
- **`must_contain` - главный инструмент оценки полноты.** Для вопроса перечисляем факты,
  без которых ответ бесполезен (нужная роль, обязательный шаг). Модели «срезают углы»
  именно здесь: на первом A/B gemma пропускала создание справочника изображений и роль
  для отгрузки - без этого инструкция невыполнима. Добавляйте `must_contain` каждый раз,
  когда замечаете такой пропуск - проверка станет ловить его автоматически.
- **Санити-прогон промпта до сравнения.** Первый A/B пришлось повторять: он вскрыл
  дефекты промпта (markdown у gemma, односложные ответы у qwen), а не разницу моделей.
  Прогоняйте новый промпт на одной модели, прежде чем сравнивать.
- **Прогоны запускать по одному.** Две команды в одной строке PowerShell срываются:
  первый процесс виснет на выходе (torch), второй не стартует. При странностях -
  `Get-Process | Where-Object { $_.Path -like "*usprings_rag*" } | Stop-Process -Force`.

Итог фиксируем в `open-questions.md` (модель, данные сравнения, обоснование) и меняем
`OPENROUTER_MODEL` в `.env` / `.env.example`.

## 6. Смена модели эмбеддингов или параметров чанкинга

Сходства разных моделей эмбеддингов **не сравнимы между собой** - одно и то же
число у другой модели значит другое. Параметры чанкинга меняют состав чанков и
сдвигают распределения.

1. Пересобрать базу: `TRUNCATE documents CASCADE` + полный ingest **каждой
   коллекции** (вопрос и чанки обязаны кодироваться одной моделью - см.
   `how-it-works.md`).
2. Полная калибровка с нуля (раздел 4) - **для каждой коллекции своя**, прежние
   значения порогов не переносятся.
3. Смена модели - через `EmbeddingProvider` + `EMBEDDING_MODEL`/`EMBEDDING_DIM`
   в `.env`; при смене размерности - миграция типа `vector(N)` и HNSW-индекса.

## 7. Мониторинг в эксплуатации

Лучшее сходство каждого запроса пишется в лог (с этапа 5, вместе с `collection`) -
реальные вопросы пользователей это лучший eval, чем синтетический набор. Именно
из этого лога пополняется `eval/questions.yaml`: текущие наборы малы (по 16-18
вопросов) и служат стартовой калибровкой, а не окончательной оценкой (см. раздел 4).

- Периодичность обзора: после первых ~100 реальных вопросов пилота, далее
  раз в месяц-два.
- **На первом же обзоре** отобрать из лога вопросы обоих типов - прошедшие порог и
  получившие отказ - и добавить в `eval/questions.yaml` соответствующей коллекции
  (с `expected_doc`, где он есть). Это первое пополнение синтетического набора
  живыми формулировками; после него пороги перепроверить (раздел 4).
- Сигналы к пересмотру порога (в логе есть `collection` - смотреть по каждой отдельно):
  - легитимные вопросы кучкуются у порога коллекции или получают отказ;
  - жалобы «мне отказали, а инструкция есть» - разобрать конкретный вопрос:
    порог это или качество поиска (recall);
  - частые ответы «информации нет» на прошедших порог вопросах - кандидаты
    на новые инструкции (пробелы покрытия) либо, при системности, на реранкер.
- Пост-MVP: структурное логирование в БД с привязкой к пользователю и аналитика
  (см. `overview.md`) - обзор станет запросом к БД вместо чтения логов.

### Разбор жалоб на ответы (после внедрения кнопки «ответ неверный», пост-MVP)

Пользователь сможет пометить ответ как неправильный; отметка пишется в лог вместе с
вопросом, ответом и диагностикой (лучшее сходство, найденные документы, ID модели).
Это самый ценный входной сигнал - реальные жалобы точнее любого синтетического набора.

Регламент разбора: по каждой жалобе определить, **где сломалось**, - три места лечатся
по-разному:

| Что видно в логе жалобы | Где сломалось | Что делать |
|-|-|-|
| Найдены не те документы | Поиск | Чанкинг, `TOP_K`, реранкер (раздел 5 недостаточен - смотреть eval, раздел 4) |
| Инструкция есть, но был отказ | Порог | Перекалибровать (раздел 4) |
| Документы верные, ответ плохой | Генерация | Промпт или смена модели (раздел 5) |
| Ответ верный, но инструкции нет в базе | Покрытие | Добавить инструкцию (раздел 1) |

Каждую подтверждённую жалобу добавлять в `eval/questions.yaml` (с `must_contain`, если
модель «срезала угол») - тогда исправление проверяется прогоном, а регресс не вернётся
незамеченным.

## 8. Заведение пользователей и доступ к коллекциям (super-admin)

Портал закрыт входом (MVP1). Пользователями и доступом управляет super-admin из UI
(`/admin/users`); ручная правка БД не нужна.

1. **Первый вход.** На чистой БД при старте создаётся один super-admin из
   `SUPERADMIN_LOGIN`/`SUPERADMIN_PASSWORD` (`.env`). После первого входа эти
   переменные из окружения можно убрать; пароль сменить в профиле (`/profile`).
2. **Создать пользователя** (`/admin/users` -> «Новый пользователь»): логин, ФИО,
   роль. При роли `user` автоматически выдаётся доступ на **все активные на тот
   момент** коллекции (не wildcard - новые коллекции доступ не получают
   автоматически, доступ к ним выдаётся отдельно). Роли:
   - `user` - задаёт вопросы по доступным коллекциям;
   - `collection_admin` - по своим коллекциям грузит/архивирует инструкции, смотрит
     журнал и аналитику; доступ к коллекциям выдаётся вручную (автогранта нет);
   - `super_admin` - весь портал, грантов не требует.
3. **Выдать/снять доступ** к коллекции - панель доступа пользователя
   (`user_collection_access`, replace-all чекбоксами). Для `collection_admin` это и
   есть назначение его коллекций.
4. **Сброс пароля** - кнопка выдаёт временный пароль (показывается один раз),
   пользователь меняет его в профиле. **Вкл/выкл** (`is_active`) - выключение
   доступа без удаления; удаления пользователя из UI нет (защита от потери истории).
   Самодеактивация запрещена.

Порог сходства коллекции, её создание и калибровка - разделы 3-4; доступ - единица

  Dockerfile/.dockerignore пересобирать без кеша или проверять содержимое образа

  `usprings_rag` до INFO (как CLI); под uvicorn без этого диагностика (старт воркера,
