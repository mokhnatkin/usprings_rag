### 2026-07-11. Этап 4 - Ingest: эмбеддинги и запись в БД (в работе)
- Статус: в работе
- Что сделано (код): `embeddings.py` - интерфейс `EmbeddingProvider` (`embed_texts`,
  `embed_query`) + `BGEEmbeddingProvider` (ленивая загрузка BGE-m3, нормализация
  векторов). `ingest/pipeline.py` - `ingest_file`: парсинг -> чанкинг -> батч-
  векторизация -> запись `documents`+`chunks` в одной транзакции; идемпотентность по
  паре (source_path, content_hash): без изменений пропускаем, при изменении удаляем
  старый документ (каскад) и вставляем заново. `cli.py` - реальный ingest по
  умолчанию, `--dry-run` под флагом. `sentence-transformers` добавлен в зависимости.
- Заблокировано загрузкой весов, приёмка ещё не пройдена.
- Инфраструктурные заметки (важно, канал ~100 КБ/с, флаки):
  - Установка `sentence-transformers` тянет torch (cp314, колесо ~116 МБ). uv не
    умеет докачивать и рвался у конца скачивания torch -> бесконечные рестарты.
    Обход: скачали колесо torch напрямую `curl -C -` (докачка), проверили sha256,
    поставили `uv pip install <локальное колесо>`, затем `uv pip install
    sentence-transformers` дотянул остальной ML-стек. torch удовлетворён.
  - ВАЖНО про запуск: `uv run` по умолчанию делает авто-sync, а sync строго сверяет
    torch с lock (источник PyPI) и переустанавливает -> снова качает torch. Поэтому
    все команды запускаем как `uv run --no-sync ...` (иначе uv уходит качать torch).
  - Веса BGE-m3: у репозитория BAAI/bge-m3 НЕТ `model.safetensors`, веса в
    `pytorch_model.bin` (~2,27 ГБ / 2271145830 байт, sha256
    b5e0ce3470abf5ef3831aa1bd5553b486803e83251590ab7ff35a117cf6aad38). Мелкие файлы
    (config, tokenizer.json, sentencepiece, 1_Pooling) уже в HF-кэше. Крупный файл
    качаем `curl -C -` прямо в snapshot-папку HF-кэша (докачка переживает обрывы;
    hf-xet и стандартный hf-resume рвались/не возобновляли). При ~100 КБ/с - ~6 часов.
  - Модель грузим один раз при старте; кэш - стандартный HF (`~/.cache/huggingface`),
    для docker позже вынесем в volume (см. mvp-dev-plan этап 8).

### 2026-07-11. Этап 3 - Ingest: парсинг и чанкинг
- Статус: сделано
- Что сделано: пакет `ingest` - `pdf.py` (постраничное извлечение через pypdf,
  dataclass `Page`), `chunker.py` (секционное разбиение по абзацам с перекрытием,
  деление слишком длинных абзацев по предложениям + жёсткий рез как крайний случай,
  привязка page_from/page_to), `cli.py` (`uv run ingest [dir]` - сухой прогон со
  статистикой, без записи в БД). Точка входа `ingest` в `pyproject.toml`.
- Проверка приёмки: на 8 PDF текст читаемый, склеек нет, пустых страниц нет; число
  чанков разумное - 7 коротких документов дают 1-2 чанка, «Планирование отгрузки»
  (17 стр.) - 14 чанков; страницы проставлены. 5 unit-тестов чанкера (`tests/
  test_chunker.py`) зелёные (упаковка, перекрытие, деление длинных, диапазон страниц).
- Решения/заметки:
  - Шаг 3.2 (запасной `pdfplumber`) не понадобился: pypdf извлёк даже крупный файл
    со сложной вёрсткой и таблицами читаемо. Зависимость не добавляли (не
    переусложняем без доказанной необходимости).
  - Размер чанка меряем оценкой токенов (эвристика ~3 символа/токен, `CHARS_PER_TOKEN`),
    не точным токенизатором BGE-m3 - достаточно для контроля размера на сухом прогоне
    и не тянет тяжёлую модель на этап 3. Точный подсчёт при необходимости - на калибровке.
  - Первая версия давала чанки-выбросы 800-1100 токенов на страницах без пустых строк
    (титул, ОГЛАВЛЕНИЕ): исходный блок склеивался в один абзац. Добавили деление
    длинного абзаца по предложениям - выбросы ушли, чанки в пределах ~300-590 токенов.
  - Колонтитулы «Страница N из N» и подписи «Рисунок N» остаются в тексте (минорный
    шум, без OCR картинки не извлекаются) - при необходимости чистим на калибровке.
  - pytest добавлен как dev-зависимость (`[dependency-groups] dev`).

### 2026-07-10. Этап 2 - Схема БД и миграции (SQLAlchemy + Alembic)
- Статус: сделано
- Что сделано: ORM-модели `Document`, `Chunk` (`models.py`), тип `vector` через
  `pgvector.sqlalchemy.Vector`; `db.py` - engine + `SessionLocal`. Alembic настроен
  (`alembic.ini`, `alembic/env.py` берёт URL и metadata из приложения). Миграции:
  `0001` - расширение vector + таблицы, `0002` - HNSW-индекс (`vector_cosine_ops`,
  m=16, ef_construction=64) и B-tree по `collection`.
- Проверка приёмки: `alembic upgrade head` создаёт расширение, таблицы и индексы
  (проверено в psql); `downgrade base` -> `upgrade head` повторяемы; `alembic check`
  сообщает "No new upgrade operations detected".
- Правки по ходу:
  - `alembic.ini` читается configparser в локальной кодировке Windows (cp1252) -
    убрал кириллицу из ini (комментарий на английском), иначе UnicodeDecodeError.
  - `created_at` в модели явно `DateTime(timezone=True)`, чтобы совпасть с `timestamptz`
    в миграции (иначе `alembic check` видел расхождение типа).
