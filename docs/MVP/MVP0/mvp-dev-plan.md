# Детальный план разработки MVP RAG

Развёрнутый инженерный план поверх `mvp-plan.md`. Тот задаёт стек и крупные этапы;
здесь - конкретные шаги, артефакты, критерии приёмки и ключевые RAG-решения. Код не
приводим - только что и в каком порядке делать, как проверять.

Принципы (из `CLAUDE.md`): инкрементально, маленькими проверяемыми шагами; не
переусложнять и не программировать защитно; каждый этап заканчивается проверкой,
без неё не идём дальше.

## 0. Ориентиры и допущения

- Объём пилота: 8 коротких PDF (1-3 стр.), текстовый слой хороший, OCR не нужен.
  Порядок величин - десятки-сотни чанков. Всё влезает в память, GPU не нужен.
- Один пользователь-тестировщик за раз, без нагрузки. Оптимизации под конкуренцию
  и масштаб не делаем, но не закрываем к ним путь (поле `collection`, абстракция
  провайдера эмбеддингов, вынос параметров в конфиг).
- Язык контента и UI - русский. Модель эмбеддингов и LLM выбраны под мультиязычность.
- ORM: используем **SQLAlchemy + Alembic** с самого начала. Таблиц сейчас мало, но
  проект пост-MVP разрастётся (логирование, аналитика, несколько баз знаний), поэтому
  единая модель данных и миграции окупаются сразу и избавляют от переписывания.
- LLM: на разработке и MVP используем только **бесплатные** модели OpenRouter (см.
  `open-questions.md`). Платный тир - вопрос прода (приватность, лимиты).

Прогресс реализации по этому плану ведём в отдельном документе
`mvp-dev-plan-progress.md`. Открытые вопросы, всплывающие по ходу, фиксируем в
`open-questions.md` (не «додумываем» молча).

### Выбор модели Claude для этапов 5-8 (рекомендация)

- **Opus 4.8 - рабочая лошадка** (можно fast mode): этапы 6, 7 и кодовая часть
  этапа 5 (5.1-5.3). Это протоптанные задачи с узким пространством решений;
  детальный план с критериями приёмки снимает основную сложность - разница
  между моделями здесь почти не проявляется.
- **Fable 5 - точечно**, там где нужна интерпретация данных и диагностика,
  а не исполнение спецификации:
  - калибровка 5.4-5.6: анализ распределений сходств relevant/irrelevant,
    выбор порога, диагностика при плохом разделении (порог - главный риск
    проекта, см. раздел 9);
  - этап 8 при неочевидных проблемах Docker/torch/Windows: не тратить
    итерации на цикл правдоподобных гипотез, переключаться на Fable для
    поиска корневой причины (проект уже собрал коллекцию таких граблей -
    см. журнал прогресса).
- Качество результата на этих этапах больше зависит от дисциплины приёмки
  (прогонять критерии каждого этапа, не идти дальше без проверки), чем от
  выбора модели.

## 1. Целевая структура репозитория

```
usprings_rag/
  pyproject.toml            # uv-проект, зависимости
  .env / .env.example       # секреты и параметры (уже есть)
  docker-compose.yml        # db + app
  Dockerfile                # образ приложения
  README.md
  src/usprings_rag/
    config.py               # загрузка настроек из .env (pydantic-settings)
    db.py                   # SQLAlchemy engine + session factory
    models.py               # ORM-модели (SQLAlchemy declarative): Document, Chunk
    embeddings.py           # интерфейс EmbeddingProvider + реализация BGE-m3
    ingest/
      pdf.py                # извлечение текста из PDF
      chunker.py            # разбиение на чанки
      pipeline.py           # оркестрация: PDF -> чанки -> эмбеддинги -> БД
      cli.py                # точка входа `uv run ingest`
    retrieval.py            # векторный поиск top-k + порог
    llm.py                  # клиент OpenRouter, сбор промпта, вызов
    answer.py               # сценарий вопрос-ответ (поиск -> порог -> LLM)
    api.py                  # FastAPI: endpoints + отдача шаблона
    templates/index.html    # экран вопрос-ответ (Jinja2)
    static/                 # минимальный JS/CSS
  alembic/                  # миграции схемы (Alembic)
  alembic.ini
  eval/
    questions.yaml          # golden-набор вопросов для калибровки
    run_eval.py             # прогон метрик поиска/ответа
  tests/                    # точечные проверки критичных мест
```

Держим модули короткими, имена ясными. `answer.py` - тонкая оркестрация; вся логика
в специализированных модулях.

## 2. Конфигурация (единый источник - `.env`)

Все настраиваемые параметры - через `config.py` (pydantic-settings), не хардкодим.
Минимальный набор ключей на MVP:

| Ключ | Назначение | Дефолт |
|-|-|-|
| `OPENROUTER_API_KEY` | ключ OpenRouter | - (обязателен) |
| `OPENROUTER_MODEL` | ID LLM | `qwen/qwen3-next-80b-a3b-instruct:free` |
| `OPENROUTER_BASE_URL` | эндпоинт | `https://openrouter.ai/api/v1` |
| `DATABASE_URL` | строка подключения Postgres | из compose |
| `EMBEDDING_MODEL` | модель эмбеддингов | `BAAI/bge-m3` |
| `EMBEDDING_DIM` | размерность | `1024` |
| `CHUNK_MAX_TOKENS` | целевой размер чанка | калибруется |
| `CHUNK_OVERLAP` | перекрытие | калибруется |
| `TOP_K` | число кандидатов поиска | калибруется |
| `SIMILARITY_THRESHOLD` | порог отказа | калибруется |
| `LLM_TEMPERATURE` | температура генерации | `0.1` (фактологичность) |
| `LLM_MAX_TOKENS` | лимит ответа | `800` |

Калибруемые параметры выносим в `.env` сразу, чтобы подбирать без правки кода.
`.env.example` держим в синхронне с этим списком.

## 3. Модель данных (детально)

Описываем как ORM-модели SQLAlchemy (declarative) в `models.py`; тип `vector`
подключаем через пакет `pgvector` (SQLAlchemy-тип `Vector` + адаптер для psycopg).
Схему разворачиваем миграциями Alembic.

`documents`
- `id` (PK)
- `collection` (text, default `it_1c`) - задел под разделение баз знаний
- `title` (text) - из имени файла / первой строки
- `source_path` (text) - путь к исходному PDF для ссылки в ответе. Хранить
  **относительным** (от корня папки инструкций) с POSIX-разделителями - иначе путь
  не переживёт перенос host -> Docker-контейнер. Сейчас пишется как передан CLI;
  нормализация - подшаг этапа 7 (перед раздачей PDF из API).
- `content_hash` (text) - хэш файла для идемпотентности повторного ingest
- `created_at` (timestamptz)

`chunks`
- `id` (PK)
- `document_id` (FK -> documents, on delete cascade)
- `chunk_index` (int) - порядок внутри документа
- `page_from`, `page_to` (int, nullable) - для точной ссылки на страницы
- `content` (text) - текст чанка
- `embedding` (vector(1024))

Индексы:
- HNSW по `embedding` с `vector_cosine_ops` (косинус - основная метрика).
- B-tree по `documents.collection` (пригодится при фильтрации по базе знаний).

Идемпотентность ingest: перед вставкой документа смотрим `content_hash`; если файл
уже загружен без изменений - пропускаем; если изменился - удаляем старый документ
(каскадом чанки) и вставляем заново. Это исключает дубли при повторном прогоне.

## 4. Ключевые RAG-решения (архитектурный слой)

Здесь - то, что определяет качество ответов. Значения подбираем на калибровке,
но подход фиксируем сейчас.

### 4.1 Извлечение текста из PDF
- Основной путь - `pypdf` (текстовый слой). Проверяем каждый из 8 файлов на
  читаемость: склейки слов, потеря переносов, порядок колонок.
- Проблемные файлы (по наблюдениям - крупные, со сложной вёрсткой, напр.
  «Планирование отгрузки») прогоняем через `pdfplumber` как запасной парсер.
- Сохраняем привязку текста к номерам страниц - для цитирования в ответе.

### 4.2 Стратегия чанкинга
- Документы короткие и структурные («шаг - скриншот - пояснение»). Базовый подход:
  разбиение по смысловым секциям (заголовки/шаги), а не слепым окном по N токенов.
- BGE-m3 держит контекст до 8192 токенов, поэтому короткий документ (1-3 стр.)
  часто помещается в 1-2 чанка целиком - это плюс для связности ответа.
- Стартовая гипотеза: чанк = логическая секция; при переполнении - деление с
  небольшим перекрытием (`CHUNK_OVERLAP`), чтобы не рвать шаг посередине.
- Метаданные чанка (документ, страницы, порядковый номер) обязательны - на них
  строится цитирование и отладка выдачи.

### 4.3 Эмбеддинги
- BGE-m3 (multilingual, dim 1024) через `sentence-transformers`, in-process.
- Нормализуем векторы (косинусная близость). Модель BGE-m3 не требует
  инструкционного префикса к запросу - вопрос и чанк кодируем одинаково.
- Провайдера прячем за интерфейсом `EmbeddingProvider` (методы `embed_texts`,
  `embed_query`), чтобы позже вынести в сервис/сменить на API без переработки.
- Модель грузим один раз при старте приложения и переиспользуем.

### 4.4 Поиск и порог «белого списка»
- Косинусный top-k по pgvector (`ORDER BY embedding <=> query` с HNSW).
- Внимание к семантике: `<=>` возвращает косинусное **расстояние**; сходство
  = `1 - расстояние`. `SIMILARITY_THRESHOLD` сформулирован как сходство - в
  `retrieval.py` пересчитываем, чтобы порог и логи говорили на одном языке.
- «Белый список» = порог по лучшему совпадению: если максимальная близость ниже
  `SIMILARITY_THRESHOLD` - вежливый отказ, LLM не дёргаем (экономим токены и
  исключаем галлюцинации на нерелевантном).
- В контекст LLM отдаём top-k чанков выше порога (или фиксированный top-k, если
  лучший прошёл порог) - подбираем на калибровке.
- Реранкер (cross-encoder) - пост-MVP; закладку оставляем, но на пилоте не ставим.

### 4.5 Промпт и генерация
- Системный промпт (рус.): роль ассистента по инструкциям 1С; отвечать строго по
  предоставленным фрагментам; если ответа в них нет - честно сказать, что информации
  нет, и не выдумывать; отвечать кратко и по шагам; в конце привести источники.
- Пользовательский промпт: вопрос + пронумерованные фрагменты с указанием документа
  и страниц.
- Температура низкая (`0.1`) - нужна фактологичность, не креатив.
- Источники в ответе формируем из метаданных чанков (`title`, `source_path`, страницы),
  а не просим LLM их придумать.
- Считываем `usage` из ответа OpenRouter (токены) - пригодится для будущего
  логирования (пост-MVP, см. `overview.md`), на MVP просто выводим в лог.

## 5. Поэтапный план (каждый этап - с критерием приёмки)

Нумерация соответствует `mvp-plan.md`, но каждый этап разбит на подшаги.

### Этап 1. Каркас и окружение
1.1 Инициализировать `uv`-проект (`pyproject.toml`), задать Python 3.12.
1.2 Базовые зависимости: `fastapi`, `uvicorn`, `pydantic-settings`, `sqlalchemy`,
    `alembic`, драйвер Postgres (`psycopg`), `pgvector` (SQLAlchemy-тип + адаптер),
    `pypdf`, `sentence-transformers`, `openai` (клиент для OpenRouter).
1.3 `docker-compose.yml`: сервис `db` (образ с pgvector), volume для данных,
    проброс порта; переменные из `.env`.
1.4 `config.py`: загрузка настроек, падение с понятной ошибкой при отсутствии
    обязательных ключей.
- Приёмка: `docker compose up db` поднимает БД; из приложения выполняется
  `CREATE EXTENSION IF NOT EXISTS vector` и `SELECT` версии расширения.

### Этап 2. Схема БД и миграции
2.1 ORM-модели `Document`, `Chunk` в `models.py`; `db.py` - engine и session factory.
2.2 Инициализировать Alembic; в первой миграции - `CREATE EXTENSION vector`, затем
    таблицы `documents`, `chunks` (генерация из моделей, тип `Vector` из `pgvector`).
2.3 Отдельной ревизией - HNSW-индекс по `embedding` (`vector_cosine_ops`) и B-tree
    по `collection`.
- Приёмка: `alembic upgrade head` на чистой БД создаёт расширение, таблицы и индексы;
  `downgrade`/`upgrade` повторяемы; модели и схема согласованы (`alembic check`).

### Этап 3. Ingest - парсинг и чанкинг
3.1 `pdf.py`: извлечение текста + номеров страниц; ручная проверка на 8 файлах.
3.2 Для проблемных файлов - ветка `pdfplumber`.
3.3 `chunker.py`: секционное разбиение с перекрытием; параметры из конфига.
3.4 `cli.py` (`uv run ingest <dir>`): обход папки, для каждого PDF - парсинг и
    чанкинг, пока без записи в БД (сухой прогон с выводом статистики).
- Приёмка: на 8 примерах текст читаемый (без грубых склеек), число чанков разумное
  (визуальная проверка нескольких чанков), страницы проставлены.

### Этап 4. Ingest - эмбеддинги и запись в БД
4.1 `embeddings.py`: интерфейс + реализация BGE-m3; загрузка модели один раз.
4.2 `pipeline.py`: батч-векторизация чанков, запись `documents` и `chunks` в одной
    транзакции на документ; идемпотентность по `content_hash`.
4.3 Подключить запись к `cli.py` (полный прогон ingest).
- Приёмка: после `uv run ingest` у всех чанков заполнен `embedding` размерности 1024;
  число документов = 8; повторный прогон не создаёт дублей.

### Этап 5. Семантический поиск и калибровка порога

_Модель: 5.1-5.3 - Opus; 5.4-5.6 (калибровка) - Fable (см. раздел 0)._

5.1 `retrieval.py`: `embed_query` -> top-k по косинусу -> вернуть чанки со
    **сходством** (`1 - расстояние`, см. 4.4) и метаданными.
5.2 Временный CLI/скрипт для ручной проверки выдачи по вопросу; лучшее сходство
    каждого запроса логируем (копим картину распределения, см. раздел 7).
5.3 Применение порога: пометка «ниже порога / выше порога».
5.4 `eval/questions.yaml`: 15-25 вопросов трёх видов - явно покрытые инструкциями
    (с ожидаемым документом), смежные/перефразированные, заведомо нерелевантные.
5.5 `eval/run_eval.py`: recall@k по покрытым вопросам + распределения сходств
    relevant vs irrelevant.
5.6 Выбор `SIMILARITY_THRESHOLD` по разделению распределений; значение и метрику
    зафиксировать в `.env` и `open-questions.md`.
- Приёмка: на контрольных вопросах релевантные чанки стабильно в топе (recall@k
  измерен); порог выбран по данным eval, а не гаданием - нерелевантные вопросы
  ниже порога, релевантные выше.

Обоснование порядка: приёмка этапа 6 («отказ на нерелевантных») напрямую зависит
от порога - калибруем его до генерации, пока это дёшево (весь корпус - десятки
чанков, LLM не участвует).

### Этап 6. Генерация ответа

_Контекст на старте (итог этапа 5, 2026-07-12):_
- Порог откалиброван: `SIMILARITY_THRESHOLD=0.53` в `.env`. Развилка
  «отказ | генерация» готова в `retrieval.py`: `SearchResult.passed` (вердикт по
  лучшему совпадению) и `SearchResult.relevant` (чанки выше порога - контекст LLM).
- Нюанс приёмки: «отказ без вызова LLM» проверять **вне-доменными** вопросами
  (погода, кухня, HR - сходства <= 0.52). Околодоменные непокрытые 1С-вопросы
  (напр., «инвентаризация склада в 1С», ~0.55) проходят порог **сознательно** -
  честное «информации нет» на них обязан дать промпт; включить такой вопрос в
  проверку 6.2, LLM-ответ «в инструкциях этого нет» - ожидаемое поведение, не
  дефект. Обоснование - `open-questions.md` (решение от 2026-07-12).
- Запуск на этой машине: все команды - `uv run --no-sync ...` (иначе uv уходит
  пере-качивать torch), русский вывод CLI - с `PYTHONUTF8=1`; ручная проверка
  поиска - `uv run --no-sync search "вопрос"` (детали - журнал прогресса).

6.1 `llm.py`: клиент OpenRouter (OpenAI SDK + base_url), сбор промпта из шаблона,
    вызов, разбор ответа и `usage`. Валидация `OPENROUTER_API_KEY` при создании
    клиента (понятная ошибка сразу, а не глубоко внутри первого запроса).
6.2 `answer.py`: сценарий поиск -> порог -> (отказ | генерация) -> ответ + источники.
6.3 Endpoint `POST /ask` в `api.py`; прогрев модели эмбеддингов в lifespan-хуке
    FastAPI (иначе первый запрос пользователя платит десятки секунд за загрузку).
6.4 A/B-сравнение LLM (см. 6.5 раздела калибровки): те же вопросы на `qwen3-next`
    и `gemma-4-31b`, фиксация выбора в `open-questions.md`.
- Приёмка: на релевантных вопросах - осмысленные пошаговые ответы со ссылками на
  верные PDF; на нерелевантных - вежливый отказ без вызова LLM (порог уже
  откалиброван на этапе 5).

### Этап 7. Портал
7.1 `templates/index.html`: поле вопроса, кнопка, область ответа, блок источников.
7.2 Минимальный JS: `fetch` к `/ask`, рендер ответа и ссылок, состояние «думаю».
7.3 Отдача статики и шаблона из FastAPI.
7.4 Раздача исходных PDF: нормализовать `source_path` в БД до относительного
    POSIX-пути от корня папки инструкций (правка `pipeline.py` + повторный ingest),
    отдавать файлы из FastAPI (StaticFiles или endpoint по `document_id`); ссылки
    в ответе ведут на этот URL, а не на путь в файловой системе.
    Нюанс: идемпотентность ingest ключуется по `source_path` - после нормализации
    старые записи не совпадут с новыми, и повторный прогон вставит дубли рядом с
    устаревшими. Поэтому перед повторным ingest очистить старые записи
    (`TRUNCATE documents CASCADE`; база пересобирается за минуты, ~23 чанка).
7.5 (опционально, первое улучшение после приёмки) Стриминг ответа через SSE:
    LLM-генерация доминирует в задержке (секунды), показ текста по мере генерации
    снижает воспринимаемое время ответа до 1-2 с до первых слов. OpenRouter
    (`stream=true`) и FastAPI (`StreamingResponse`) поддерживают штатно; отказ
    по порогу остаётся мгновенным обычным ответом.
- Приёмка: сквозной сценарий в браузере - вопрос -> ответ со ссылками; отказ виден
  как отдельное состояние; ссылки открывают исходный PDF из браузера.

### Этап 8. Упаковка и запуск с нуля

_Модель: Opus; при неочевидных проблемах Docker/torch/Windows - Fable (см. раздел 0)._

8.1 `Dockerfile` приложения. Обязательно: torch ставить **CPU-колесом**
    (`--index-url https://download.pytorch.org/whl/cpu`) - дефолтная установка в
    Linux тянет CUDA-сборку на несколько ГБ, бесполезную без GPU.
8.2 В compose - сервис `app` рядом с `db`. Монтируем в `app`: (а) HF-кэш хоста
    (`~/.cache/huggingface`) - веса BGE-m3 (2,27 ГБ) уже скачаны, повторная
    загрузка на нашем канале - часы; (б) папку инструкций `docs/manuals/` - для
    ingest и раздачи PDF (относительные `source_path` совпадают на хосте и в
    контейнере).
8.3 Порядок старта: `db` -> миграции -> `app`; прогон ingest документирован как
    `docker compose run app ingest` (или эквивалент).
8.4 README: запуск (`docker compose up`), прогон ingest, где взять ключ OpenRouter,
    список калибруемых параметров.
- Приёмка: на чистой машине `docker compose up` + прогон ingest дают рабочий портал
  без ручных доводок.

## 6. Калибровка и оценка качества

Без измеримой калибровки порог и параметры - гадание. Заводим лёгкий eval-контур.
Калибровка не отдельный «этап в конце», а встроена в последовательность: поисковая
часть (6.1-6.3) выполняется в составе этапа 5 (подшаги 5.4-5.6), до генерации;
A/B по LLM (6.5) - в составе этапа 6.

6.1 `eval/questions.yaml`: 15-25 вопросов трёх видов - явно покрытые инструкциями,
    смежные/перефразированные, заведомо нерелевантные (для проверки отказа). Для
    покрытых - ожидаемый исходный документ. _(= подшаг 5.4)_
6.2 `run_eval.py`: прогоняет вопросы через поиск, считает попадание нужного документа
    в top-k (recall@k) и распределение близостей relevant vs irrelevant. _(= 5.5)_
6.3 Порог `SIMILARITY_THRESHOLD` выбираем по разделению распределений: максимум
    отсечения нерелевантного при сохранении релевантного. Фиксируем метрику и
    значение. _(= 5.6)_
6.4 Итерации по `CHUNK_MAX_TOKENS`, `CHUNK_OVERLAP`, `TOP_K` - меняем в `.env`,
    перегоняем eval, сравниваем. Результаты и выбранные значения фиксируем в
    `open-questions.md` (раздел калибровок). По мере надобности на этапах 5-6.
6.5 A/B по LLM: те же вопросы на `qwen3-next` и `gemma-4-31b`, сравнение качества
    ответов на русском и следования инструкции «только по контексту». _(= 6.4)_

## 7. Наблюдаемость (минимум на MVP)

- Структурный лог на запрос: вопрос (усечённо), число кандидатов, лучшая близость,
  вердикт (ответ/отказ), выбранная модель, токены из `usage`, время ответа.
- Лучшую близость логируем начиная с этапа 5 (ещё до LLM) - к моменту калибровки
  и далее в эксплуатации копится картина распределения сходств.
- Это же - подготовка к пост-MVP логированию с привязкой к пользователю и аналитике
  (см. `overview.md`); на MVP без ID пользователя и без хранения в БД.

## 8. Definition of Done (MVP)

- `docker compose up` на чистой машине поднимает БД и портал.
- `uv run ingest` наполняет базу из `docs/manuals/IT_1C/` идемпотентно.
- На контрольном наборе: релевантные вопросы - ответы по существу со ссылками на
  верные PDF; нерелевантные - вежливый отказ.
- Порог и параметры поиска подобраны на eval и зафиксированы в `.env` и документации.
- README позволяет развернуть и прогнать всё без устной передачи знаний.

## 9. Риски и на что смотреть

- Качество текстового слоя отдельных PDF (крупные файлы со сложной вёрсткой) -
  проверить на этапе 3, при необходимости `pdfplumber`.
- Слишком мелкий чанк рвёт пошаговые инструкции; слишком крупный размывает поиск -
  балансируем на калибровке (документы короткие - вероятен крупный чанк).
- Порог: узкий разброс близостей relevant/irrelevant усложнит «белый список» -
  тогда рассмотреть реранкер (пост-MVP) или связку порог + число прошедших чанков.
- Лимиты и приватность бесплатного тира OpenRouter - для калибровки приемлемо,
  для прода платный тир с zero-data-retention (см. `open-questions.md`).
- Первый старт тянет веса BGE-m3 - предусмотреть кэш модели в volume.
