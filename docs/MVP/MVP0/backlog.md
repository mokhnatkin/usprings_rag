## Изоляция поиска (решение 5, обоснование)

**Проблема.** С приблизительными индексами (HNSW) фильтр применяется *после*
обхода индекса - это прямо сказано в README pgvector. При `ef_search=40` и
коллекции, покрывающей 10% корпуса, до фильтра в среднем доживают 4 строки из 40:
поиск «недобирает» релевантное, потому что top-k съедают соседние базы.

**Ключевой нюанс нашей схемы.** Сейчас `collection` лежит в `documents`, а
HNSW-индекс - на `chunks`. Фильтр по джойну индексу вообще не виден: он
отрабатывает уже по результату векторного обхода. Поэтому при любом варианте
`collection` **денормализуется в `chunks`** - фильтр обязан стоять на той же
таблице, где вектор.

**Выбранный вариант - секционирование** (`chunks PARTITION BY LIST (collection)`,
HNSW-индекс в каждой секции). Планировщик по `WHERE collection = 'erp'` отсекает
лишние секции (partition pruning) и идёт в индекс только нужной коллекции:
недобора нет по построению, скорость не зависит от размера соседних баз. README
pgvector рекомендует именно секционирование, когда значений фильтра много -
а у нас коллекции будут множиться.

Отклонённые альтернативы:
- *Частичные индексы* (`CREATE INDEX ... WHERE collection = 'erp'`) - изоляция
  тоже полная и схема меняется меньше, но pgvector советует их для небольшого
  числа значений; при росте до десятков коллекций всё равно придётся переезжать
  на секции, а переезд с обычной таблицы на секционированную - это перезалив данных.
- *Общий индекс + `hnsw.iterative_scan`* - схему почти не трогает, но поиск
  по-прежнему обходит чужие векторы (лишняя работа растёт с числом коллекций), а
  при сильном перекосе возможен недобор из-за `hnsw.max_scan_tuples`.

Стоимость выбранного варианта: PK у `chunks` становится составным (`id, collection` -
требование Postgres: ключ секционирования обязан входить в PK); добавление
коллекции требует `CREATE TABLE ... PARTITION OF` + `CREATE INDEX` (автоматизируется
в ingest, а в пост-MVP - в админке при создании коллекции).

### B. Данные и схема

- **B1. Ввести enum коллекций в коде.** Одно место (например `collections.py`):
  код (`erp`, `zup`), отображаемое название («1С:ERP», «1С:ЗУП»), папка инструкций
  (`its_erp`, `its_zup`), порог сходства. Используется ingest, поиском, API и UI.
- **B2. Секционированная схема `chunks` (решение 5).** Миграция Alembic:
  `chunks.collection` (NOT NULL), составной PK `(id, collection)`,
  `PARTITION BY LIST (collection)`, секции `chunks_erp` / `chunks_zup`, HNSW-индекс
  в каждой секции (`m=16, ef_construction=64, vector_cosine_ops` - как сейчас).
  `collection` в `documents` остаётся (метаданные документа), в `chunks` -
  денормализованная копия, проставляется из документа при ingest.
  Создание секции при появлении новой коллекции - в ingest (`CREATE TABLE IF NOT
  EXISTS ... PARTITION OF`), чтобы добавление коллекции не требовало ручной DDL.
  **Данные при миграции не переносятся:** `chunks` пересоздаётся пустой, `documents`
  очищается. Текущие строки с `collection='it_1c'` не легли бы ни в одну секцию
  (`erp`/`zup`), а корпус всё равно пересобирается в B4 - переносить нечего.
  Приёмка: `EXPLAIN` на поисковом запросе показывает обращение к одной секции
  (partition pruning) и Index Scan по её HNSW-индексу, без Seq Scan по соседям;
  таблицы documents/chunks пусты.
- **B3. Заполнять `collection` при ingest.** `pipeline.ingest_file` принимает
  коллекцию, пишет её в `Document` и в каждый `Chunk`; идемпотентность остаётся по
  `source_path`. `ingest/cli.py`: обязательный `--collection`, убрать `DEFAULT_DIR =
  docs/manuals/IT_1C` (папки больше не будет) - папку брать из enum по коллекции.
  Тесты `test_pipeline.py:14-20` (фикстуры с путями `IT_1C/...`) правятся здесь же:
  сигнатура `ingest_file` меняется, откладывать починку до блока E - значит жить
  с красными тестами между инкрементами.
- **B4. Пересобрать данные ERP под коллекции.** Порядок: удалить корпус
  `docs/manuals/IT_1C/` (решение 2); переименовать `docs/manuals/parsed/` ->
  `docs/manuals/its_erp/`; прогнать ingest с `--collection erp` на новой схеме
  (БД к этому моменту пуста - очистку сделала миграция B2). Дефолт `it_1c` в
  `models.py:20` убрать. Стоимость: переиндексация 411 PDF - около 1,5 часов.

## Порядок выполнения

A1 -> A2 -> (A3 идёт фоном, это часы) -> B1 -> B2 -> B3 -> B4 -> C1..C4 -> D1..D2 ->
E1..E6 -> F. B5 (индексация ЗУП) встраивается, как только готовы A3 и B3; от него
зависят E3, E4 в части порога `zup` и E5. Блоки B, C, D и тест изоляции (E1) не
зависят от наличия PDF ЗУП - делаются параллельно с прогоном A3. Пороги (E4) можно
зафиксировать только после пересборки eval (E2/E3), поэтому до тех пор поиск
работает на временном значении. Тесты чинятся вместе с изменением сигнатур
(B3/C1/C2, см. E6) - между инкрементами прогон остаётся зелёным.

Важно: миграция схемы (B2) идёт **до** переиндексации (B4) - иначе полтора часа
эмбеддингов зальются в таблицу, которую миграция тут же пересоздаст пустой.

**Что идёт не по коду, а по данным:** идущая сейчас (2026-07-14) индексация 411
PDF ERP в коллекцию `it_1c` - промежуточная. Её снесёт миграция схемы (B2), и в B4
она будет повторена уже с `--collection erp` из папки `its_erp/`. Отдельного смысла
доводить её до конца нет, но и вреда нет - она подтверждает, что корпус
индексируется.

## Риски

- **R1. Секционирование даёт побочные ограничения Postgres.** Ключ секционирования
  обязан входить в PK (`chunks.id` -> `(id, collection)`); уникальность `id` сама по
  себе больше не гарантируется БД. Пересечение коллекций в одном запросе (поиск
  «сразу по двум базам») перестаёт быть бесплатным - но по решению владельца такого
  сценария нет: пользователь всегда выбирает одну коллекцию.
- **R2. Объём ЗУП неизвестен.** Оценка появится после A1 (по числу строк реестра).
  Для ERP: 3645 строк -> 411 PDF -> 800 МБ -> около 10 часов парсинга и 1,5 часа
  индексации.
- **R3. Провал в измеримости качества.** С удалением старого корпуса (решение 2)
  исчезает единственный измеренный ориентир (recall@1 = 17/17 на пороге 0.53).
  Пока не собран новый golden-набор (E2), качество поиска на корпусе ИТС
  **неизвестно** - это главный технический долг блока.
- **R4. Расползание `source_path`.** Переименование папок инструкций порождает
  сиротские строки в БД (ingest сверяет по пути). Чистить осознанно (очистка -
  миграцией B2, пересборка - B4), иначе поиск будет ссылаться на несуществующие
  файлы.
