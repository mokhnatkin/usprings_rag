# Backlog MVP0: вторая коллекция (1С:ЗУП) и разделение баз знаний

Анализ и задачи. Код не написан - это план работ.

Задача от владельца (2026-07-14): добавить второй источник инструкций -
`https://its.1c.ru/db/zupcorpdoc` (1С:ЗУП). Инструкции ЗУП и ERP хранить в
**разных коллекциях**; пользователь в UI выбирает коллекцию до вопроса; поиск

из соседней базы. На MVP0 коллекции - enum в коде; справочник коллекций с

## Риски

- **R1. Фильтрация + HNSW.** Векторный индекс ищет ближайших по всей таблице, а
  `WHERE collection=...` отсекает результаты **после** индекса: при сильном перекосе
  размеров коллекций top-k может «съесться» соседней базой и вернуть меньше k
  релевантных. В pgvector 0.8 для этого есть итеративное сканирование
  (`hnsw.iterative_scan`), альтернатива - частичные индексы по коллекциям.
  Проверить на реальных данных (E5), решение зафиксировать в open-questions.
- **R2. Объём ЗУП неизвестен.** Оценка появится после A1 (по числу строк реестра).
  Для ERP: 3645 строк -> 411 PDF -> 800 МБ -> около 10 часов парсинга и 1,5 часа
  индексации.
- **R3. Провал в измеримости качества.** С удалением старого корпуса (решение 2)
  исчезает единственный измеренный ориентир (recall@1 = 17/17 на пороге 0.53).
  Пока не собран новый golden-набор (E2), качество поиска на корпусе ИТС
  **неизвестно** - это главный технический долг блока.
- **R4. Расползание `source_path`.** Переименование папок инструкций порождает
  сиротские строки в БД (ingest сверяет по пути). Чистить осознанно (B3), иначе
  поиск будет ссылаться на несуществующие файлы.
