# Ориентировочный план развёртывания MVP RAG

Черновой план: стек и этапы для минимального работающего пилота. Дополняет
`docs/overview.md` и `docs/open-questions.md`. Решения по эмбеддингам, скриншотам
и загрузке приняты как ориентиры для MVP (не финальные - см. раздел «Калибровки»).

## Что делаем на MVP

Веб-портал с одним экраном «вопрос-ответ» поверх базы знаний из инструкций 1С.
Без авторизации. Наполнение базы - CLI-скриптом по PDF из `docs/manuals/IT_1C/`.
Ответ строится только при наличии релевантного фрагмента выше порога, иначе -
вежливый отказ.

## Наблюдения по исходным инструкциям

Разобраны примеры из `docs/manuals/IT_1C/` (8 PDF):

- У PDF есть нормальный **текстовый слой** - вся инструктивная проза извлекается
  как текст. OCR для MVP не требуется.
- Встроены скриншоты интерфейса 1С (часто с красными пометками). Действия
  продублированы текстом рядом со скриншотом, поэтому для поиска и ответа
  достаточно текста.
- Документы короткие (1-3 страницы), структура «шаг - скриншот - пояснение».

Вывод: на MVP индексируем текстовый слой, скриншоты не обрабатываем, но в ответе
даём ссылку на исходный PDF, чтобы пользователь мог посмотреть картинки.

## Стек MVP

| Область | Выбор |
|-|-|
| Язык / пакеты | Python 3.12, `uv` |
| Бэкенд / API | FastAPI + uvicorn |
| Векторное хранилище | PostgreSQL 16 + pgvector (косинус, HNSW-индекс) |
| Парсинг PDF | `pypdf` (текстовый слой; при проблемах с версткой - `pdfplumber`) |
| Эмбеддинги | Локальная multilingual-модель BGE-m3 (dim 1024) через `sentence-transformers`, in-process |
| LLM | OpenRouter (через OpenAI-совместимый SDK с base_url OpenRouter) |
| Фронтенд | Jinja2-шаблон + минимальный JS (fetch) |
| Инфраструктура | Docker Compose, on-premise |

Заметки:
- Эмбеддинги на MVP держим в процессе приложения (без отдельного сервиса) - проще.
  Интерфейс провайдера эмбеддингов абстрагируем, чтобы позже вынести в сервис или
  сменить на API без переработки пайплайна.
- CPU-инференс BGE-m3 для пилотного объёма приемлем; GPU не обязателен.

## Схема данных (минимум)

- `documents` - `id`, `title`, `source_path`, `created_at`.
- `chunks` - `id`, `document_id`, `chunk_index`, `content`, `embedding vector(1024)`.
- Заранее закладываем поле `collection` (или `scope`) в `documents` со значением
  по умолчанию `it_1c`. На MVP одно значение; позже - разделение баз знаний по
  подразделениям без миграции структуры (см. `overview.md`, масштабирование).

## Поток «вопрос-ответ»

1. Векторизуем вопрос той же моделью эмбеддингов.
2. Поиск top-k ближайших чанков в pgvector (косинусное сходство).
3. Проверка порога по лучшему совпадению: ниже порога - вежливый отказ.
4. Иначе собираем промпт с найденными чанками, вызываем LLM через OpenRouter.
5. Возвращаем ответ и ссылки на исходные документы (`source_path`).

## Этапы (инкрементально, с проверкой каждого)

1. **Каркас и окружение.** Структура репозитория, `uv`-проект, `.env`,
   `docker-compose` с Postgres + pgvector.
   _Проверка:_ контейнер БД поднимается, расширение `vector` доступно.
2. **Схема БД.** Таблицы `documents`, `chunks`, векторный индекс.
   _Проверка:_ таблицы и индекс создаются, миграция повторяема.
3. **Ingest: парсинг и чанкинг.** CLI `uv run ingest`: чтение PDF из папки,
   извлечение текста, разбиение на чанки (по абзацам/секциям с перекрытием).
   _Проверка:_ на 8 примерах корректный текст и разумное число чанков.
4. **Ingest: эмбеддинги.** Подключить BGE-m3, векторизовать чанки, записать в БД.
   _Проверка:_ у всех чанков заполнен `embedding` правильной размерности.
5. **Семантический поиск.** Endpoint/скрипт поиска top-k по вопросу.
   _Проверка:_ на тестовых вопросах релевантные чанки в топе выдачи.
6. **Генерация ответа.** OpenRouter: промпт с контекстом, ответ + источники,
   порог отказа для «белого списка».
   _Проверка:_ осмысленные ответы на релевантные вопросы, отказ на нерелевантные.
7. **Портал.** Простая страница вопрос-ответ (Jinja2 + fetch к API).
   _Проверка:_ сквозной сценарий в браузере.
8. **Упаковка.** `docker-compose up` поднимает всё с нуля; краткий README запуска
   и инструкция по прогону ingest.
   _Проверка:_ чистый запуск на пустой машине.

## Вне MVP (пост-MVP, заложить возможность)

- **Авторизация:** внутренняя + LDAP / Microsoft AD (архитектуру не ломать).
- **Админ-UI загрузки** инструкций вместо CLI.
- **Разделение баз знаний по подразделениям** (поле `collection` уже заложено).
- **Скриншоты:** OCR или мультимодальное описание для поиска по картинкам.
- **React-фронтенд**, сбор обратной связи, логирование запросов и качества ответов.

## Калибровки (подобрать на прототипе)

- Размер чанка и перекрытие (документы короткие - возможно, секция целиком).
- `top-k` для поиска.
- Порог сходства для отказа (метрика + значение).
- Конкретная LLM-модель на OpenRouter (баланс качество/цена, поддержка русского).
