Накопитель задач на будущее (после MVP1). Здесь - направления и решения владельца,
не детальный dev-plan. Заводим по мере появления задач, чтобы не потерять.

MVP1 закрыт: многопользовательский портал (аутентификация внутренними учётками,

## Задача 1. Разделить доступ collection-admin: «управление» и «вопросы»

**Проблема.** Сейчас доступ к коллекции - одна связка `user_collection_access`
(пара `user_id` x `collection_id`), а её смысл зависит от роли: для `user` - право
задавать вопросы, для `collection_admin` - право администрировать И спрашивать
(`auth.check_collection_access(..., need_admin=True/False)` читает ту же таблицу).
Значит collection-admin может задавать вопросы только по тем же коллекциям, которыми
управляет, - а на практике круг «где могу спросить» должен быть **шире**, чем «куда
могу загружать документы».

**Чего хочет владелец.**
- У `collection_admin` **два независимых набора коллекций**:
  - **управление** (узкий) - загрузка/архивация документов, запуск индексации;
    только те коллекции, которые ему назначены как администратору;
  - **вопросы** (широкий) - по какому набору он может спрашивать; шире набора
    управления.
- **При создании пользователя по умолчанию** доступ «задавать вопросы» - **на все
  коллекции** (не только на активные на момент создания, как сейчас автогрант для
  `user`).

**Текущее состояние (на что опираемся).**
- `user_collection_access` - одна таблица-связка, UNIQUE `(user_id, collection_id)`.
- `auth.accessible_codes` - вычисляет доступные коллекции; `check_collection_access`
  с `need_admin` - гейт управляющих операций (этап 7 MVP1, `admin/documents.py`).
- `collections_service.create_user` - автогрант `user` на **все активные на тот
  момент** коллекции (снимок, не wildcard; новые коллекции доступ не получают).
- Изоляция поиска - на уровне секций `chunks`; доступ - отдельный слой прав.

**Открытые решения (уточнить у владельца перед реализацией, не «додумывать»).**
- **Модель данных для двух наборов.** Варианты:
  - колонка-признак в `user_collection_access` (напр. `scope`/`can_manage`) - тогда
    «вопросы на все» = либо строка на каждую коллекцию (снимок, новые не появятся
    автоматически), либо отдельная семантика «все»;
  - «вопросы» как **широкое правило по умолчанию** (все активные коллекции, динамично)
    с возможностью сузить, а «управление» - явными грантами; тогда таблица грантов
    хранит только управление, а ask-доступ - по умолчанию все, минус исключения.
  Рекомендация: ask-доступ по умолчанию = все активные коллекции (динамично, чтобы
  новые коллекции сразу были доступны для вопросов), manage-доступ = явные гранты.
  Требует ли пилот вообще сужать ask-доступ у отдельных пользователей - решить: если
  нет, слой ask-грантов можно не заводить.
- **`super_admin`** по-прежнему видит всё (грантов не требует) - без изменений.
- **`user`** (не админ): ask-доступ по правилу по умолчанию (все); management ему не
  нужен.

**Правки (ориентир).**
- Развести в `auth` два запроса доступа: `ask_codes(user)` (широкий) и
  `manage_codes(user)` (узкий, только для админов); `check_collection_access`
  разбить на `require_ask_access` и `require_manage_access`.
- `/ask`, `/ask/stream`, `/collections` - по ask-доступу; `/admin/documents*`
  (загрузка/архивация/индексация) - по manage-доступу.
- `admin/users` - раздельная выдача: назначение управляемых коллекций (админам) и,
  если решим сужать, ask-доступа.
- Миграция под выбранную модель данных; обновить тесты доступа
  (`test_access.py`, `test_admin_documents.py`).

**Приёмка.**
- `collection_admin` грузит/архивирует документы только в назначенных ему
  (управляемых) коллекциях - в остальных 403; при этом задаёт вопросы по более
  широкому набору (все коллекции по умолчанию).
- Новый пользователь по умолчанию может задавать вопросы по всем коллекциям
  (включая созданные после него, если принято динамическое правило).
- Изоляция поиска по секциям не меняется; управляющие операции по-прежнему за
  серверными гейтами (403), а не только скрытием в меню.

## Задача 2. Аутентификация через LDAP / Microsoft Active Directory

**Контекст.** MVP1 - только внутренние учётки (логин+пароль, argon2, cookie-сессии).
Провайдер аутентификации намеренно спрятан за тонким интерфейсом (`auth.authenticate`),
чтобы LDAP/AD добавился **второй реализацией** без переработки потока входа, сессий и
RBAC. Это давно зафиксированное пост-MVP направление (см. `docs/open-questions.md`).

**Цель.** Пользователи входят на портал доменными учётными записями (Microsoft AD),
пароли на стороне AD, портал их не хранит. Роли и доступ к коллекциям - по членству в
доменных группах (маппинг группа -> роль/коллекции) либо, на первом шаге, по-прежнему
ведутся в приложении.

**Решения владельца (приняты 2026-07-21).**
- **Оба способа входа одновременно:** внутренние учётки остаются (страховка -
  bootstrap-super-admin и сервисные аккаунты при недоступном AD), доменные
  добавляются вторым провайдером.
- **Выбор провайдера - автоматически, без переключателя в UI:** одна форма
  логин+пароль; если логин заведён как локальный - проверяем локально, иначе (и если
  LDAP включён) - через AD.
- **В портал пускаем всех сотрудников домена** - ограничивать вход отдельной группой
  безопасности не будем.
- **Роли не берём из групп AD.** При первом входе доменный пользователь создаётся с
  минимальной ролью `user`; роль и доступ к коллекциям при необходимости
  переназначает super-admin в портале (`/admin/users`). Маппинг групп - отдельная
  тема на будущее, доменные группы под роли заводить не просим.
- **Провижининг - JIT:** запись в БД создаётся при первом успешном входе; профиль
  (ФИО, почта) обновляется из AD при последующих входах.
- **Способ проверки пароля:** прямой bind учётными данными пользователя по LDAPS
  (предпочтительно); сервисный bind - запасной вариант, если прямой запрещён политикой.

**Что запросить у администратора AD (до начала работ).**

Готовый опросник для пересылки администраторам - `docs/MVP/MVP2/ldap-questions.md`,
разделён на обязательную часть A (без неё работу не начинаем) и желательную часть B,
у каждой свой чек-лист «параметр -> значение».

- **Часть A (блокирует старт):** адреса контроллеров домена и порт (636 LDAPS
  предпочтительно, либо 389 + StartTLS), правило firewall с хоста приложения,
  CA-сертификат домена, имена домена (DNS и NetBIOS); разрешён ли прямой user-bind
  (иначе - DN и пароль сервисной read-only учётки); базовый DN поиска, атрибут логина
  (`sAMAccountName` или `userPrincipalName`), атрибуты профиля (`displayName`/`cn`,
  `mail`); тестовая доменная учётка.
- **Часть B (уточняет поведение):** фильтр только включённых учёток, одинаковые логины
  в разных доменах леса, политики блокировки и паролей, поведение при увольнении,
  заведомо отключённая тестовая учётка, контакт по сбоям и окна техработ, согласование
  ИБ, допустимость кэша профиля в БД портала.

**Правки (ориентир).**
- `models.py`: у `User` - `auth_source` (`local` | `ldap`, default `local`),
  `password_hash` nullable (у доменных пуст). Миграция `0009`.
- `config.py`: блок LDAP-настроек (`ldap_enabled`, серверы, LDAPS/CA, base DN,
  атрибут логина, фильтр, при необходимости сервисный bind, группа доступа); значения
  в `.env` / `.env.example`.
- Новый `ldap_auth.py` на `ldap3` (`uv add ldap3`): проверка пароля bind-ом и чтение
  профиля; наружу - только успех/неуспех и профиль, пароль не логируется.
- `auth.authenticate` становится диспетчером локальный/LDAP + JIT-создание записи с
  ролью `user`. Смена и сброс пароля недоступны для `auth_source=ldap`.
- Тесты с подменённым LDAP-провайдером (без реального AD).
- (Следующим шагом) маппинг доменных групп -> роль и доступ к коллекциям.

**Приёмка.**
- Доменный пользователь входит своей учёткой AD (пароль в портале не хранится),
  получает сессию и работает по своим правам; отключённая в AD учётка не входит.
- Внутренний вход для bootstrap-super-admin продолжает работать (страховка при
  недоступности AD).
- Приватность: пароль не логируется и не сохраняется; трафик к AD - по TLS.
