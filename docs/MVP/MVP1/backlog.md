# Backlog MVP1: пользователи, роли, логирование, история, админка

## Цель MVP1

Превратить открытый пилот в многопользовательский портал с разграничением
доступа, журналом вопросов-ответов, историей для пользователя и админ-интерфейсом
управления коллекциями, пользователями и инструкциями. MVP0 (коллекции,
секционирование, пороги) закрыт и остаётся фундаментом.

Задача от владельца (2026-07-15), шесть направлений:

1. Аутентификация и авторизация пользователей.
2. Управление ролями: `user`, `collection-admin`, `super-admin`.
3. Логирование вопросов и ответов в привязке к пользователю и коллекции
   (поля - в `docs/overview.md`, раздел логирования).
4. История вопросов для пользователя (последние 3 на главной + полная с пагинацией).
5. Справочники коллекций и пользователей для super-admin.
6. Интерфейс загрузки инструкций / пометки «неактуальные» для collection-admin
   и super-admin.

## Роли (модель прав)

- **`user`** - задаёт вопросы и получает ответы по коллекциям, к которым имеет
  доступ. При создании пользователя ему выдаётся доступ на **все существующие на
  тот момент** коллекции (явные гранты, не wildcard - новые коллекции доступ не
  получают автоматически).
- **`collection-admin`** - администратор одной или нескольких коллекций. По своим
  коллекциям: загружает новые инструкции и помечает старые неактуальными,
  запускает индексацию, смотрит лог вопросов-ответов и аналитику. Может задавать
  вопросы по своим коллекциям.
- **`super-admin`** - создаёт пользователей с любыми ролями, выдаёт
  collection-admin доступ к коллекциям, запускает калибровку порогов, смотрит лог
  и аналитику по всему порталу, ведёт справочники коллекций и пользователей.
  При развёртывании бутстрапится **одна** учётка super-admin.

У пользователя одна роль. Доступ к коллекциям - через таблицу-связку
`user_collection_access`; super-admin её не требует (видит всё).

## Принятые решения (владелец, 2026-07-15)

1. **Аутентификация - только внутренние учётки.** Логин + пароль (хеш),
   cookie-сессии. LDAP/Microsoft AD откладываем на пост-MVP, но архитектуру не
   ломаем под будущее добавление внешнего провайдера. Bootstrap super-admin - из
   переменных окружения при первом старте.
2. **Коллекции переезжают из enum в БД.** Таблица `collections`
   (`code`, `title`, `folder`, `threshold`, ...). Super-admin создаёт и
   редактирует коллекции через UI; создание коллекции заводит секцию `chunks`
   (+ HNSW-индекс) и папку инструкций. ingest, поиск, API и UI читают справочник
   из БД. `collection.py` (enum + dataclass) становится источником начальных
   данных для миграции-сида, дальше - read-model поверх БД.
3. **Загрузка инструкций - PDF, single-file ingest, soft-delete.** Файл
   загружается через UI в папку коллекции, индексируется по одному. «Неактуальные»
   = soft-delete (`documents.archived_at`): документ и его чанки исключаются из
   поиска, но остаются в БД для аудита и истории. Word (docx) - пост-MVP.
4. **Аналитика - базовые агрегаты.** Число запросов и токенов за период, доля
   отказов, доля помеченных «неверными», активность по часам/дням, топ вопросов.
   Простые таблицы и графики. Разрез: коллекция (collection-admin) и портал
   (super-admin). Развёрнутый дашборд - пост-MVP.

Решения по остаточным вопросам (владелец, 2026-07-15):

5. **Индексация загруженного PDF - фоновая задача со статусом.** Загрузка ставит
   задачу в очередь, UI показывает статус (в очереди / индексируется / готово /
   ошибка). Требует таблицу задач и воркер (см. A6, H2). Синхронный вариант
   отклонён - минутная индексация не должна блокировать админа и мешать пакетной
   загрузке.
6. **Возврат документа из архива разрешён.** Админ может снять `archived_at` и
   вернуть документ в поиск - при soft-delete это почти бесплатно (чанки остаются
   в БД) и страхует от случайной архивации (см. H3).
7. **Пользователь сам меняет свой пароль** (старый + новый) в профиле; super-admin
   по-прежнему может сбросить пароль на временный (см. B6, I1).
8. **У отметки «ответ неверный» - опциональное поле комментария**
   (`feedback_comment`), помогает при разборе жалоб (см. A5, F).

Умолчания (приняты как разумные, поправимо):

- **Хеш пароля** - argon2 (`pwdlib`), либо `passlib[bcrypt]`, если argon2 не встанет.
- **Сессия** - подписанная cookie (Starlette `SessionMiddleware`, `SECRET_KEY` в
  `.env`), HttpOnly + SameSite. Серверных сессий в БД не заводим (пилотный масштаб).
- **CSRF** - для state-changing форм админки (double-submit token или SameSite=Strict).
- **Логика фильтра `archived_at`** - джойн-фильтр в поиске (`documents.archived_at
  IS NULL`); денормализовать в `chunks` - только если появится заметный недобор
  top-k (см. R5).

## Что уже есть в коде (факты, на что опираемся)

- `collection.py` - enum `CollectionCode` (`erp`, `zup`), dataclass `Collection`
  (`code`, `title`, `folder`, `threshold`), `COLLECTIONS`, `get_collection`.
  Единственный справочник, потребители - ingest, `retrieval.search`, `api`, UI.
- `models.py` - `Document` (есть `collection`, `content_hash`, `created_at`),
  `Chunk` (секционирован `PARTITION BY LIST (collection)`, составной PK
  `(id, collection)`, HNSW-индекс на родителе).
- `retrieval.search` (`retrieval.py:79`) принимает `Collection`, фильтрует по
  `Chunk.collection`, порог берёт из `collection.threshold`. Джойнит `Document`
  за метаданными - **сюда добавится фильтр `archived_at`**.
- `answer.py` - `answer_question` / `stream_answer` возвращают `Answer`
  (`text`, `refused`, `sources`, `best_similarity`, `elapsed_seconds`).
  **Токены и `model` в `Answer` не протянуты**, хотя `llm.py` их уже отдаёт.
- `llm.py` - `LLMResponse` уже несёт `prompt_tokens`, `completion_tokens`,
  `model`; стрим шлёт usage последним чанком (`include_usage=True`, usage сейчас
  только логируется, наружу не возвращается).
- `api.py` - `/ask`, `/ask/stream`, `/collections`. Аутентификации нет,
  пользователь не известен. `AskRequest`/`AskResponse` - без user/лога.
- `config.py` - настройки из `.env` (pydantic-settings). Порог - НЕ здесь
  (он свойство коллекции).
- `ingest/cli.py`, `ingest/pipeline.py` - ingest по папке коллекции, идемпотентен
  по `source_path`, только PDF.
- Фронтенд - `templates/index.html` + статика, один экран вопрос-ответ, без
  роутинга и без сессии.

## Архитектурные заметки

**Коллекции в БД (решение 2) - это рефактор горячего пути.** Секции `chunks`
привязаны к строковому `collection` (коду). Значит:
- `collections.code` остаётся стабильным строковым ключом (им же назван раздел
  секции). Меняем справочник и его чтение, схему секционирования - нет.
- `retrieval.search`, `answer_*`, `api` работают с объектом-коллекцией, у которого
  есть `.code` и `.threshold`. Достаточно, чтобы read-model из БД повторяла
  интерфейс нынешнего dataclass `Collection` - тогда правки в поиске минимальны.
- Создание коллекции из UI должно атомарно: строка в `collections` + `CREATE TABLE
  ... PARTITION OF chunks` + папка инструкций. Секцию уже умеет заводить ingest
  (`CREATE TABLE IF NOT EXISTS ... PARTITION OF`) - переиспользуем.

**Токены для лога уже почти готовы.** `LLMResponse` несёт usage; надо протянуть
их (и `model`) в `Answer`, а в стриме - вернуть usage из финального чанка вместе с
событием `done`. Отдельная интеграция не нужна.

**Лог не должен ронять ответ.** Запись в журнал оборачиваем так, чтобы сбой
логирования не срывал пользовательский запрос (простой `try/except` вокруг insert,
без переусложнения).

**Полнота текста в истории (решение владельца 2026-07-15).** Храним **полный**
вопрос и **полный** ответ - этого требуют история (пункт 4) и переиспользование
жалоб для eval. `overview.md` формулировал «первые N символов» - это расхождение
снимается: N (env) остаётся только как усечение для списков и аналитики, не для
хранения. Актуализировать `overview.md` в блоке M1.

## Задачи

### A. Схема данных и миграции

- **A1. Таблица `collections`.** Миграция Alembic: `id`, `code` (UNIQUE, NOT NULL),
  `title`, `folder`, `threshold`, `is_active`, `created_at`. Сид из текущего
  `collection.py` (`erp` -> 0.58, `zup` -> 0.55, папки `its_erp`/`its_zup`).
  Приёмка: миграция создаёт таблицу и заливает две коллекции с текущими порогами.
- **A2. Таблица `users`.** `id`, `login` (UNIQUE), `full_name`, `email` (опц.),
  `password_hash`, `role` (enum: `user` | `collection_admin` | `super_admin`),
  `is_active`, `created_at`.
  Приёмка: таблица создаётся; роль ограничена enum на уровне БД или приложения.
- **A3. Таблица `user_collection_access`.** Связка `user_id` x `collection_id`,
  UNIQUE-пара. Для `user` - право задавать вопросы; для `collection_admin` -
  право администрировать (и спрашивать). `super_admin` грантов не требует.
  Приёмка: гранты создаются/снимаются, дубли невозможны.
- **A4. Soft-delete документов.** `documents.archived_at` (nullable timestamp) +
  `archived_by` (опц., FK `users`). Индекс под фильтр активных.
  Приёмка: миграция добавляет колонки, существующие документы - активные.
- **A5. Таблица `query_log`.** Поля (по `overview.md` + история + диагностика):
  `id`, `user_id` (FK), `collection_id` (FK), `created_at`, `question` (полный),
  `answer` (полный), `outcome` (`answered` | `refused`), `best_similarity`,
  `prompt_tokens`, `completion_tokens`, `total_tokens`, `elapsed_seconds`,
  `model_id`, `sources` (JSON: id/названия найденных документов - для разбора
  жалоб), `feedback` (nullable: помечен неверным), `feedback_at`, `feedback_comment`
  (опц.). Индексы: `(user_id, created_at)`, `(collection_id, created_at)`.
  Приёмка: таблица создаётся; вставка тестовой записи проходит.
- **A6. Таблица задач индексации `index_job`.** `id`, `collection_id` (FK),
  `document_id` (FK, nullable до создания документа), `source_path`,
  `status` (`queued` | `running` | `done` | `error`), `error` (текст, nullable),
  `created_by` (FK `users`), `created_at`, `started_at`, `finished_at`. Под
  фоновую индексацию (решение 5).
  Приёмка: таблица создаётся; статус ограничен перечислением.

### B. Аутентификация

- **B1. Хеширование паролей.** Модуль `security.py`: `hash_password`,
  `verify_password` (argon2). `uv add pwdlib` (или `passlib[bcrypt]`).
  Приёмка: хеш и проверка работают, юнит-тест на round-trip.
- **B2. Сессии и текущий пользователь.** `SessionMiddleware` (`SECRET_KEY` в `.env`,
  HttpOnly, SameSite). Зависимость `get_current_user` (по `user_id` в сессии),
  `require_role(...)` / `require_collection_access(...)`. 401 для неавторизованных,
  403 при нехватке прав.
  Приёмка: защищённый эндпоинт без сессии даёт 401, с чужой ролью - 403.
- **B3. Логин/логаут.** `GET /login` (форма), `POST /login` (проверка,
  установка сессии), `POST /logout`. Неактивный пользователь (`is_active=false`)
  войти не может.
  Приёмка: верные креды пускают, неверные - нет, логаут рвёт сессию.
- **B4. Bootstrap super-admin.** При старте: если пользователей нет - создать
  super-admin из `.env` (`SUPERADMIN_LOGIN`, `SUPERADMIN_PASSWORD`). Идемпотентно
  (повторный старт учётку не плодит и пароль молча не меняет).
  Приёмка: чистая БД -> один super-admin; повторный старт -> без дублей.
- **B5. Закрыть портал за авторизацией.** `/`, `/ask`, `/ask/stream` требуют
  сессию. `AskRequest`/`/ask/stream` больше не принимают произвольную коллекцию
  без проверки доступа (см. C2).
  Приёмка: аноним на `/` редиректится на `/login`.
- **B6. Смена собственного пароля** (решение 7). Экран профиля: старый пароль +
  новый (с проверкой старого). Действует для любой роли.
  Приёмка: с верным старым паролем смена проходит, с неверным - отказ; новый
  пароль работает при следующем входе.

### C. Авторизация в потоке вопрос-ответ

- **C1. Список коллекций - по доступу.** `/collections` возвращает только
  коллекции, к которым у текущего пользователя есть доступ (super-admin - все
  активные).
  Приёмка: `user` с доступом только к `erp` не видит `zup`.
- **C2. Проверка доступа в `/ask` и `/ask/stream`.** Коллекция берётся из БД;
  если у пользователя нет доступа - 403 (не молчаливый поиск). `get_collection`
  из enum заменяется чтением из справочника (блок D).
  Приёмка: вопрос в недоступную коллекцию -> 403.

### D. Коллекции из БД (рефактор справочника)

- **D1. Read-model коллекции из БД.** Загрузка коллекций из `collections` с тем же
  интерфейсом, что у нынешнего dataclass `Collection` (`.code`, `.title`,
  `.folder`, `.threshold`). `get_collection(code)` читает БД. Кэш в памяти с
  инвалидизацией при правке (пилотный масштаб - можно читать на запрос).
  Приёмка: `retrieval.search` и `answer_*` работают без изменения сигнатур;
  тесты поиска зелёные.
- **D2. Потребители переведены на БД-справочник.** `api.py` (`/collections`,
  `/ask*`), `ingest/cli.py` (папка и порог из БД), eval-скрипты. Enum
  `CollectionCode` остаётся только для валидации/сида, не как источник правды.
  Приёмка: `grep` не находит бизнес-логику, зависящую от `COLLECTIONS` как от
  источника истины (только сид/валидация).
- **D3. Создание коллекции заводит секцию и папку.** Сервис `create_collection`:
  строка в `collections` + `CREATE TABLE ... PARTITION OF chunks` (переиспользовать
  код ingest) + создание папки `folder`. Откат при сбое любого шага.
  Приёмка: новая коллекция сразу пригодна для ingest и поиска; секция и папка есть.

### E. Логирование вопросов-ответов

- **E1. Протянуть usage и model в `Answer`.** `answer_question` кладёт
  `prompt_tokens`/`completion_tokens`/`model` из `LLMResponse` в `Answer`;
  `stream_answer` возвращает usage из финального чанка в событии `done`. У отказа
  токены = 0 (LLM не звали).
  Приёмка: `Answer` несёт токены и модель; для отказа - нули.
- **E2. Запись в `query_log` на каждый ответ.** После `/ask` и завершения
  `/ask/stream` пишем строку: пользователь, коллекция, полный вопрос/ответ,
  outcome, best_similarity, токены, elapsed, model, sources. Сбой записи не роняет
  ответ. `QUERY_LOG_PREVIEW_CHARS` (env) - усечение для списков/аналитики.
  Приёмка: после вопроса в БД появляется корректная строка; при искусственном
  сбое insert пользователь всё равно получает ответ.
- **E3. Отказ тоже логируется.** Запрос ниже порога и `NO_ANSWER` в контексте -
  `outcome=refused`, best_similarity записан (по нему смотрим поведение порога на
  реальных вопросах - см. `open-questions.md`).
  Приёмка: отказ даёт строку лога с `outcome=refused`.

### F. Обратная связь по ответу («ответ неверный»)

Поле `feedback` есть в списке логируемых (`overview.md`), поэтому входит в MVP1.

- **F1. Кнопка «ответ неверный»** в портале под ответом + опциональное поле
  комментария (решение 8).
- **F2. Эндпоинт отметки.** `POST /feedback` по `query_log.id` текущего
  пользователя: ставит `feedback`, `feedback_at`, `feedback_comment`.
  Приёмка: отметка сохраняется в ту же строку лога; чужую запись пометить нельзя.

### G. История вопросов пользователя

- **G1. Последние 3 вопроса на главной.** Блок под формой: три последних записи
  `query_log` текущего пользователя (вопрос + время + признак отказа), клик -
  переход к полному ответу.
  Приёмка: показываются ровно последние три записи пользователя.
- **G2. Просмотр одного вопроса-ответа.** `GET /history/{id}` (только своя
  запись): полный вопрос, полный ответ, источники, время.
  Приёмка: своя запись открывается, чужая - 403/404.
- **G3. Полная история с пагинацией.** `GET /history` - список записей
  пользователя, страницами (`page`/`page_size`), новые сверху; ссылка с главной.
  Приёмка: пагинация работает, порядок по убыванию времени.

### H. Админка: инструкции (collection-admin, super-admin)

- **H1. Список документов коллекции.** По доступным коллекциям: документы с
  статусом (активен / архивирован), датой, числом чанков.
  Приёмка: collection-admin видит документы только своих коллекций.
- **H2. Загрузка PDF + фоновая индексация одного файла** (решение 5). Форма
  загрузки: файл кладётся в папку коллекции, создаётся `index_job` (A6), воркер
  прогоняет single-file ingest в секцию коллекции; UI показывает статус задачи
  (в очереди / индексируется / готово / ошибка). Идемпотентность по `source_path`
  сохраняется. Воркер - в рамках приложения (фоновый исполнитель без внешнего
  брокера, см. R7).
  Приёмка: после загрузки задача проходит статусы до `done`, PDF появляется в
  документах и находится поиском; при ошибке парсинга задача - `error` с текстом.
- **H3. Пометка «неактуальные» (soft-delete) и возврат из архива** (решение 6).
  Кнопка архивации: проставляет `documents.archived_at` (+ `archived_by`), поиск
  исключает архивные. Кнопка возврата: снимает `archived_at`, документ снова в
  поиске (переиндексация не нужна - чанки на месте).
  Приёмка: архивированный документ пропадает из выдачи, остаётся в списке как
  архивный и в старых логах; возврат возвращает его в выдачу.
- **H4. Фильтр архивных в поиске.** `retrieval.search` добавляет
  `documents.archived_at IS NULL`.
  Приёмка: чанки архивных документов не попадают в выдачу.

### I. Админка: справочники (super-admin)

- **I1. Справочник пользователей.** Список, создание пользователя с ролью,
  включение/выключение (`is_active`), сброс пароля. Выдача collection-admin
  доступа к коллекциям (правка `user_collection_access`). При создании `user` -
  автогрант на все активные коллекции (решение по ролям).
  Приёмка: super-admin создаёт пользователей всех трёх ролей и управляет доступом.
- **I2. Справочник коллекций.** Список, создание (через сервис D3), правка
  `title`/`threshold`/`is_active`. Удаление коллекции - вне MVP1 (риск по данным),
  только деактивация.
  Приёмка: super-admin создаёт коллекцию из UI, она сразу доступна для ingest и
  поиска; правит порог.
- **I3. Просмотр лога вопросов-ответов.** super-admin - по всему порталу,
  collection-admin - по своим коллекциям. Список с фильтром по коллекции/периоду,
  провал в запись (вопрос, ответ, диагностика, отметка «неверно»).
  Приёмка: collection-admin видит лог только своих коллекций.

### J. Калибровка порогов (super-admin)

- **J1. Запуск калибровки по коллекции.** Кнопка «пересчитать порог»: прогоняет
  eval по golden-набору коллекции (`eval/run_eval.py`), отдаёт распределения
  сходств и **рекомендованный** порог. Долгий прогон - фоновая задача со статусом.
  Приёмка: по коллекции возвращается рекомендованный порог с обоснованием (min
  релевантных / max негативов), как в записях калибровки 2026-07-14.
- **J2. Применение порога.** super-admin сохраняет рекомендованный (или свой)
  порог в `collections.threshold` - без деплоя.
  Приёмка: изменённый порог сразу действует в поиске (D1).

### K. Аналитика (базовые агрегаты)

- **K1. Агрегаты по логу.** Запросы (SQL) поверх `query_log`: число запросов и
  токенов за период, доля отказов, доля помеченных «неверными», активность по
  часам/дням недели, топ вопросов. Разрез: коллекция (для collection-admin - его
  коллекции) и портал (super-admin).
  Приёмка: цифры сходятся с содержимым лога на тестовой выборке.
- **K2. Экран аналитики.** Простые таблицы + графики (сервер отдаёт данные, фронт
  рисует минимально). collection-admin - свои коллекции; super-admin - портал и
  срез по пользователям.
  Приёмка: collection-admin видит аналитику только своих коллекций.

### L. UI и навигация

- **L1. Каркас с сессией и ролями.** Шапка с логином/логаутом; пункты меню по
  роли (вопрос, история; для админов - документы, лог, аналитика; для super-admin -
  пользователи, коллекции, калибровка).
  Приёмка: пункты меню соответствуют роли; чужие экраны недоступны и по прямой ссылке.
- **L2. Экран вопрос-ответ под сессией.** Селектор коллекции - из доступных (C1),
  блок истории (G1), кнопка обратной связи (F1).
  Приёмка: сквозной сценарий залогиненного пользователя работает.

### M. Документация

- **M1. `docs/overview.md`** - авторизация (внутренняя, роли), логирование
  (полный текст + N как усечение - расхождение из «Архитектурных заметок»),
  аналитика как реализованная. После E, K.
- **M2. `docs/open-questions.md`** - перенести в «Принятые» решения по auth,
  коллекциям-в-БД, soft-delete, аналитике; открытым оставить LDAP/AD и приватность LLM.
- **M3. `CLAUDE.md`** - роли, коллекции в БД (был enum), soft-delete, справочники.
- **M4. `README.md`** - запуск с bootstrap super-admin (`SUPERADMIN_*`, `SECRET_KEY`),
  загрузка инструкций через UI (CLI-ingest остаётся для массовой заливки).
- **M5. `docs/maintenance.md`** - калибровка порога из UI, добавление коллекции из

- **M6. `docs/MVP/MVP1/` dev-plan** - при желании детальный план разработки, по

## Открытые вопросы внутри MVP1

Все решены владельцем 2026-07-15 (см. «Принятые решения» ниже).

## Порядок выполнения

Фундамент -> доступ -> данные -> фичи. Рекомендуемая последовательность:

1. **A** (схема) -> **B** (auth, bootstrap) -> **D** (коллекции в БД: рефактор
   справочника, до навешивания прав на коллекции).
2. **C** (авторизация в потоке) - поверх B и D.
3. **E** (логирование) -> **F** (обратная связь) -> **G** (история) - дают
   пользовательскую ценность и данные для аналитики.
4. **H** (инструкции), **I** (справочники), **J** (калибровка) - админка.
5. **K** (аналитика) - последней, поверх накопленного лога.
6. **L** (UI/навигация) - собирается параллельно с соответствующими блоками.
7. **M** (документация) - вместе с кодом каждого блока, не в конце.

Инкрементально: каждый блок - маленькие проверяемые шаги, тесты правятся вместе с
изменением сигнатур (как в MVP0), прогон между инкрементами зелёный.

## Риски

- **R1. Рефактор коллекций в БД трогает горячий путь поиска (D).** Секции `chunks`
  завязаны на строковый `code`; ошибка в read-model или рассинхрон
  `collections` <-> секции ломает поиск. Митигируем стабильностью `code` и тем, что
  схему секционирования не трогаем - меняем только источник справочника.
- **R2. Создание коллекции из UI - многошаговая операция (D3).** Строка в БД +
  DDL секции + папка должны быть согласованы; частичный сбой оставит «полу-коллекцию».
  Нужен откат/проверка целостности.
- **R3. Токены в стриме.** usage приходит финальным чанком с пустым `choices` -
  логирующий код на `/ask/stream` должен дождаться его до записи `query_log`.
- **R4. Безопасность внутренних учёток.** Хеш пароля, HttpOnly/SameSite cookie,
  CSRF для админ-форм, отказ неактивным. Не переусложнять, но базовое закрыть.
- **R5. Soft-delete и недобор top-k.** Фильтр `archived_at` - джойн после обхода
  индекса: чанки архивного документа занимают места в top-k и отсекаются после. На
  пилотном объёме приемлемо; при заметном недоборе - денормализовать `archived`
  в `chunks` (как `collection`).
- **R6. Лог как узкое место надёжности.** Запись лога не должна ронять ответ
  (try/except вокруг insert), но и молча терять записи нежелательно - логировать
  сбой самого логирования.
- **R7. Фоновый воркер индексации на on-premise (H2).** Внешнего брокера
  (Celery/Redis) не заводим - для пилота достаточно фонового исполнителя в
  процессе приложения (например, `BackgroundTasks` / отдельный поток-пул,
  дёргающий очередь `index_job`). Риски: перезапуск контейнера теряет незавершённую
  задачу (при старте помечать «зависшие» `running` как `error`), а тяжёлый ingest
  (загрузка весов BGE-m3 уже в памяти) не должен конкурировать с ответами за
  ресурсы. Если в проде понадобится масштаб - вынести в отдельный воркер-процесс.

## Пост-MVP1 (не делаем сейчас)

- LDAP / Microsoft AD как второй провайдер аутентификации.
- Word (docx) при загрузке инструкций (парсер по расширению).
- Развёрнутый аналитический дашборд (фильтры, drill-down, экспорт).
- Реранкер для околодоменных вопросов (тянется из MVP0 open-questions).
- Удаление коллекции из UI (сейчас только деактивация - риск по данным).
- Вынос воркера индексации в отдельный процесс / внешний брокер (в MVP1 - фоновый
  исполнитель в приложении, см. H2/R7).
- Приватность LLM (zero-data-retention) - обязательна перед продом, не привязана
  к MVP1.
