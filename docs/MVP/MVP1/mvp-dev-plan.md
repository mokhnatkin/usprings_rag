# Детальный план разработки MVP1

Развёрнутый инженерный план поверх `backlog.md`. Backlog задаёт направления и
решения владельца; здесь - конкретные шаги, артефакты, критерии приёмки и
ключевые технические решения. Код не приводим - только что, в каком порядке и как
проверять.

Принципы (из `CLAUDE.md`): инкрементально, маленькими проверяемыми шагами; не
переусложнять и не программировать защитно; сначала корневая причина, потом
правка; каждый этап заканчивается проверкой, без неё дальше не идём.

MVP0 (этапы 1-8 + коллекции) - фундамент: рабочий портал вопрос-ответ, коллекции
`erp`/`zup`, секционирование `chunks`, пороги в `collection.py`. MVP1 надстраивает
над ним пользователей, роли, логирование, историю и админку.

## 0. Ориентиры и допущения

- Масштаб пилота прежний: десятки пользователей, единичная нагрузка. Оптимизации
  под конкуренцию не делаем, но путь к ним не закрываем (провайдер аутентификации
  абстрагируем под будущий LDAP/AD; воркер индексации - в приложении, с заделом на
  вынос).
- Аутентификация - **только внутренние учётки** (решение владельца). LDAP/Microsoft
  AD - пост-MVP; интерфейс аутентификации закладываем так, чтобы добавление
  внешнего провайдера не ломало поток.
- Коллекции **переезжают из enum в БД** - это фундаментный рефактор горячего пути
  поиска, поэтому идёт первым (этап 1).
- Тексты вопросов и ответов храним **полностью** (для истории и переиспользования
  жалоб в eval); N символов - только усечение для списков/аналитики.
- Загрузка инструкций - PDF, индексация **фоновой** задачей со статусом. Word (docx)
  - пост-MVP.
- Язык UI и текстов - русский (клиенты в Казахстане/РФ). Без эмодзи в коде и выводе.

Прогресс по этому плану ведём в `mvp-dev-plan-progress.md`. Открытые вопросы,
всплывающие по ходу, фиксируем в `open-questions.md` (не «додумываем» молча).

### Выбор модели Claude для этапов (рекомендация)

- **Opus 4.8 - рабочая лошадка** (можно fast mode): почти всё - схема, auth, RBAC,
  логирование, история, админка, UI. Задачи с узким пространством решений; детальный
  план с критериями приёмки снимает основную сложность.
- **Fable 5 - точечно**, где нужна интерпретация/диагностика, а не исполнение
  спецификации:
  - калибровка порогов (этап 9): анализ распределений сходств, выбор порога -
    как в MVP0 (порог остаётся чувствительным местом);
  - неочевидные проблемы сессий/CSRF/секций Postgres/воркера/Docker - не тратить
    итерации на цикл правдоподобных гипотез, переключаться на Fable для корневой
    причины.
- Качество больше зависит от дисциплины приёмки (прогонять критерии каждого этапа),
  чем от выбора модели.

## 3. Модель данных (детально)

Новые ORM-модели в `models.py`, схема - миграциями Alembic. Существующие
`documents`/`chunks` не переписываем, только добавляем `documents.archived_at`.

`collections` (справочник коллекций - переезд из enum)
- `id` (PK)
- `code` (text, UNIQUE, NOT NULL) - **стабильный** строковый код (`erp`, `zup`);
  им же назван раздел секции `chunks`. Менять код нельзя (привязка к секции).
- `title` (text) - название для UI («1С:ERP»)
- `folder` (text) - папка инструкций относительно `MANUALS_DIR` (`its_erp`)
- `threshold` (float) - порог сходства коллекции
- `is_active` (bool, default true) - деактивация без удаления
- `created_at` (timestamptz)
- Сид миграцией из текущего `collection.py`: `erp` (0.58, `its_erp`),
  `zup` (0.55, `its_zup`).

`users`
- `id` (PK)
- `login` (text, UNIQUE, NOT NULL)
- `full_name` (text)
- `email` (text, nullable)
- `password_hash` (text) - argon2
- `role` (text/enum: `user` | `collection_admin` | `super_admin`)
- `is_active` (bool, default true) - выключение доступа без удаления
- `created_at` (timestamptz)

`user_collection_access` (связка доступа)
- `user_id` (FK -> users, on delete cascade)
- `collection_id` (FK -> collections, on delete cascade)
- UNIQUE `(user_id, collection_id)`
- Семантика зависит от роли пользователя: для `user` - право задавать вопросы;
  для `collection_admin` - право администрировать коллекцию (и спрашивать по ней).
  `super_admin` записей не требует (видит всё).

`documents` (+ soft-delete)
- `archived_at` (timestamptz, nullable) - непусто = неактуальна, исключается из поиска
- `archived_by` (FK -> users, nullable)
- Индекс под фильтр активных (частичный по `archived_at IS NULL` либо обычный).

`query_log` (журнал вопросов-ответов)
- `id` (PK)
- `user_id` (FK -> users)
- `collection_id` (FK -> collections)
- `created_at` (timestamptz)
- `question` (text) - **полный** текст вопроса
- `answer` (text) - **полный** текст ответа (у отказа - текст отказа)
- `outcome` (text: `answered` | `refused`)
- `best_similarity` (float) - поведение порога на реальных вопросах
- `prompt_tokens`, `completion_tokens`, `total_tokens` (int) - у отказа 0
- `elapsed_seconds` (float)
- `model_id` (text) - какая LLM ответила
- `sources` (JSONB) - id/названия найденных документов (для разбора жалоб)
- `feedback` (bool, nullable) - помечен ли ответ неверным
- `feedback_at` (timestamptz, nullable)
- `feedback_comment` (text, nullable)
- Индексы: `(user_id, created_at desc)`, `(collection_id, created_at desc)`.

`index_job` (задачи фоновой индексации)
- `id` (PK)
- `collection_id` (FK -> collections)
- `document_id` (FK -> documents, nullable до создания документа)
- `source_path` (text) - загруженный файл
- `status` (text: `queued` | `running` | `done` | `error`)
- `error` (text, nullable) - текст ошибки при `error`
- `created_by` (FK -> users)
- `created_at`, `started_at`, `finished_at` (timestamptz, последние два nullable)

Почему `collections.code` неизменяем: секции `chunks` (`chunks_erp`, `chunks_zup`)
привязаны к строковому значению коллекции. Смена кода потребовала бы пересоздания
секции и переиндексации - в MVP1 запрещаем правку `code` (правим `title`,
`threshold`, `is_active`).

## 4. Ключевые решения (архитектурный слой)

### 4.1 Аутентификация и сессии
- Внутренние учётки: логин + пароль. Хеш - **argon2** (`pwdlib`; запасной вариант
  `passlib[bcrypt]`, если argon2 не встанет в контейнере).
- Сессия - подписанная cookie через Starlette `SessionMiddleware` (`SECRET_KEY`),
  в сессии храним `user_id`. Серверных сессий в БД не заводим (пилотный масштаб).
- Cookie: `HttpOnly`, `SameSite=Lax` (или `Strict` для админки), `Secure` за TLS.
- Провайдера аутентификации прячем за тонким интерфейсом (проверка логина/пароля),
  чтобы LDAP/AD добавился как вторая реализация без переработки потока.
- Bootstrap super-admin: при старте, если таблица `users` пуста - создать одного
  super-admin из `SUPERADMIN_*`. Идемпотентно (есть пользователи - ничего не делаем,
  пароль молча не меняем).

### 4.2 Роли и авторизация (RBAC)
- Одна роль на пользователя (`user` | `collection_admin` | `super_admin`).
- Зависимости FastAPI: `get_current_user` (по сессии; 401 если нет),
  `require_role(*roles)` (403 если роль не подходит),
  `require_collection_access(collection, admin=False)` - проверяет
  `user_collection_access` (super_admin проходит всегда; `admin=True` требует роль
  `collection_admin`+ и грант на коллекцию).
- Доступ пользователя `user` при создании: явные гранты на **все активные на тот
  момент** коллекции (не wildcard - новые коллекции автоматически не открываются).
- Ошибки прав - честные коды (401/403), не «молчаливый» поиск по общей базе.

### 4.3 Коллекции в БД (рефактор справочника)
- `collection.py` из enum-справочника превращается в **read-model** поверх таблицы
  `collections`: тот же интерфейс объекта коллекции (`.code`, `.title`, `.folder`,
  `.threshold`), что и нынешний dataclass, - тогда `retrieval.search` и `answer_*`
  меняются минимально (сигнатуры сохраняются).
- Кэш справочника в памяти с инвалидизацией при правке из админки; на пилотном
  масштабе допустимо и просто читать из БД на запрос.
- `create_collection` (сервис) - атомарно: строка в `collections` +
  `CREATE TABLE ... PARTITION OF chunks` (переиспользовать код ingest) + создание
  папки инструкций. Откат при сбое любого шага, чтобы не осталось «полу-коллекции».
- `CollectionCode` (enum) остаётся только для сида миграции и валидации входных
  значений там, где это удобно, - не как источник истины.

### 4.4 Фоновая индексация
- Загрузка PDF создаёт `index_job` со `status=queued`; воркер берёт задачу,
  проставляет `running`, гоняет single-file ingest в секцию коллекции, ставит
  `done`/`error`. UI опрашивает статус.
- Воркер - **в процессе приложения** (без внешнего брокера): фоновый исполнитель
  (поток-пул / `BackgroundTasks`), дёргающий очередь. Модель BGE-m3 уже в памяти
  (lifespan) - переиспользуем, не грузим повторно.
- Устойчивость: при старте приложения «зависшие» `running` (после аварийного
  рестарта) помечаем `error` - иначе задача застрянет навсегда. Тяжёлый ingest не
  должен конкурировать с ответами за ресурсы - ограничить параллелизм (1 задача
  за раз на пилоте достаточно).

### 4.5 Логирование вопросов-ответов
- `LLMResponse` уже несёт `prompt_tokens`/`completion_tokens`/`model` - протягиваем
  их (и модель) в `Answer`. В стриме usage приходит финальным чанком (пустой
  `choices`) - логирующий код дожидается его перед записью.
- Запись в `query_log` - после формирования ответа (и отказа тоже). Обёрнута так,
  чтобы сбой записи **не ронял** ответ пользователю (try/except вокруг insert,
  сам сбой логируем).
- Храним полный вопрос/ответ; `QUERY_LOG_PREVIEW_CHARS` - усечение в списках.
- `sources` (найденные документы), `best_similarity`, `model_id` пишем всегда -
  это диагностика для разбора жалоб (какое звено чинить: поиск/порог/генерация).

### 4.6 Soft-delete и фильтр поиска
- Архивация: `documents.archived_at = now()`, `archived_by`. Возврат: `archived_at
  = NULL` (переиндексация не нужна - чанки на месте).
- `retrieval.search` добавляет фильтр `documents.archived_at IS NULL` (джойн уже
  есть). На пилотном объёме недобор top-k из-за архивных чанков пренебрежим;
  денормализовать `archived` в `chunks` - только если станет заметно (см. риски).

## 5. Поэтапный план (каждый этап - с критерием приёмки)

Порядок: фундамент -> доступ -> данные -> пользовательские фичи -> админка ->
аналитика -> упаковка. Тесты и правки документации идут **вместе** с кодом этапа,
не в конце. Документацию (блок M бэклога) актуализируем по завершении смежных этапов.

### Этап 1. Коллекции в БД (фундамент)

Рефактор горячего пути - делаем первым и аккуратно, поиск не должен сломаться.

1.1 Модель `CollectionRow` + миграция таблицы `collections`; сид из `collection.py`
    (`erp` 0.58 / `its_erp`, `zup` 0.55 / `its_zup`).
1.2 `collection.py` -> read-model поверх БД: `get_collection(code)` и перечисление
    читают `collections`; интерфейс объекта сохранён (`.code/.title/.folder/.threshold`).
    Кэш в памяти.
1.3 Перевести потребителей на БД-справочник: `api.py` (`/collections`, `/ask*`),
    `ingest/cli.py` (папка и порог из БД), eval-скрипты. `COLLECTIONS`/enum - только
    сид/валидация.
1.4 `collections_service.create_collection`: строка + `PARTITION OF` + папка,
    атомарно, с откатом.
- Приёмка: тесты поиска (`test_retrieval`) зелёные без изменения сигнатур; поиск по
  `erp`/`zup` работает как раньше (`EXPLAIN` - обращение к своей секции); новая
  тестовая коллекция создаётся сервисом и сразу пригодна для ingest и поиска;
  `grep` не находит бизнес-логику, завязанную на enum как на источник истины.

### Этап 2. Аутентификация

2.1 Миграция `users`; модель `User`.
2.2 `security.py`: `hash_password`/`verify_password` (argon2), юнит-тест round-trip.
2.3 `auth.py`: `SessionMiddleware` (`SECRET_KEY`), `get_current_user`.
2.4 Логин/логаут: `GET /login` (форма), `POST /login` (проверка, установка сессии),
    `POST /logout`. Неактивный пользователь войти не может.
2.5 Bootstrap super-admin из `SUPERADMIN_*` при пустой `users`, идемпотентно.
2.6 Закрыть портал за сессией: `/`, `/ask`, `/ask/stream` требуют авторизации;
    аноним на `/` -> редирект на `/login`.
- Приёмка: чистая БД -> при старте один super-admin; повторный старт без дублей;
  верные креды пускают, неверные и неактивные - нет; логаут рвёт сессию; аноним не
  попадает на портал.

### Этап 3. Авторизация и доступ к коллекциям

3.1 Миграция `user_collection_access`; модель связки.
3.2 Зависимости `require_role`, `require_collection_access` (super_admin - сквозной
    проход).
3.3 `/collections` отдаёт только доступные пользователю коллекции.
3.4 `/ask` и `/ask/stream`: коллекция из БД, проверка доступа - иначе 403.
3.5 `B6` смена своего пароля: `profile.html` + эндпоинт (старый + новый пароль).
- Приёмка: `user` с доступом только к `erp` не видит и не может спросить `zup` (403);
  `collection_admin` администрирует только свои коллекции; смена пароля с верным
  старым проходит, с неверным - отказ, новый пароль работает при следующем входе.

### Этап 4. Логирование вопросов-ответов

4.1 Миграция `query_log`; модель `QueryLog`.
4.2 Протянуть usage и `model` из `LLMResponse` в `Answer`; в `stream_answer`
    вернуть usage из финального чанка в событии `done`.
4.3 `logging_qa`: запись строки лога после `/ask` и завершения `/ask/stream`
    (пользователь, коллекция, полный вопрос/ответ, outcome, best_similarity, токены,
    elapsed, model, sources). Сбой записи не роняет ответ.
4.4 Отказ (ниже порога / `NO_ANSWER`) логируется с `outcome=refused`, токены 0.
- Приёмка: после вопроса в `query_log` корректная строка (токены, сходство,
  источники); отказ даёт `outcome=refused`; при искусственном сбое insert
  пользователь всё равно получает ответ.

### Этап 5. Обратная связь по ответу

5.1 Кнопка «ответ неверный» в портале + опциональное поле комментария.
5.2 `POST /feedback` по `query_log.id` текущего пользователя: `feedback`,
    `feedback_at`, `feedback_comment`. Чужую запись пометить нельзя.
- Приёмка: отметка сохраняется в ту же строку лога; попытка пометить чужую запись
  - 403/404.

### Этап 6. История вопросов пользователя

6.1 `history`: выборки последних N и постранично, по убыванию времени.
6.2 Блок «последние 3 вопроса» на главной (вопрос + время + признак отказа), клик
    -> полный ответ.
6.3 `GET /history/{id}` - только своя запись (полный вопрос/ответ, источники, время).
6.4 `GET /history` - полная история с пагинацией (`page`/`page_size`), ссылка с главной.
- Приёмка: на главной ровно три последних записи пользователя; своя запись
  открывается, чужая - 403/404; пагинация и порядок по убыванию времени работают.

### Этап 7. Инструкции: загрузка и soft-delete

7.1 Миграции: `documents.archived_at`/`archived_by`; таблица `index_job`.
7.2 `retrieval.search` + фильтр `archived_at IS NULL`.
7.3 `indexing/jobs` + `indexing/worker`: постановка задачи, фоновый single-file
    ingest в секцию коллекции, статусы; при старте «зависшие» `running` -> `error`.
7.4 Админ-экраны документов: список по доступным коллекциям (статус/дата/чанки),
    форма загрузки PDF (создаёт `index_job`, показывает статус), архивация и возврат
    из архива.
- Приёмка: загруженный PDF проходит статусы до `done`, появляется в документах и
  находится поиском; ошибка парсинга -> `error` с текстом; архивированный документ
  пропадает из выдачи, остаётся в списке как архивный и в старых логах; возврат
  возвращает его в выдачу; collection-admin видит только свои коллекции.

### Этап 8. Справочники super-admin

8.1 `admin/users`: список, создание пользователя с ролью, вкл/выкл (`is_active`),
    сброс пароля, гранты доступа к коллекциям (`user_collection_access`).
    При создании `user` - автогрант на все активные коллекции.
8.2 `admin/collections`: список, создание (через `create_collection`), правка
    `title`/`threshold`/`is_active`. Удаление - вне MVP1 (только деактивация).
8.3 `admin/logs`: просмотр `query_log` - super-admin по порталу, collection-admin по
    своим коллекциям; фильтр по коллекции/периоду, провал в запись (вопрос, ответ,
    диагностика, отметка «неверно»).
- Приёмка: super-admin создаёт пользователей всех трёх ролей и управляет доступом;
  создаёт коллекцию из UI (сразу пригодна для ingest и поиска), правит порог;
  collection-admin видит лог только своих коллекций.

### Этап 9. Калибровка порогов (super-admin)

_Модель: Fable для интерпретации распределений (см. раздел 0)._

9.1 `admin/calibration`: запуск eval по golden-набору коллекции
    (`eval/run_eval.py`) фоновой задачей; сбор распределений сходств и
    **рекомендованного** порога (по правилу MVP0: ниже минимума релевантных с
    запасом, выше максимума негативов).
9.2 Применение: super-admin сохраняет рекомендованный (или свой) порог в
    `collections.threshold` - изменение действует в поиске сразу (read-model,
    этап 1).
- Приёмка: по коллекции возвращается рекомендованный порог с обоснованием (min
  релевантных / max негативов); сохранённый порог немедленно меняет поведение
  «белого списка».

### Этап 10. Аналитика (базовые агрегаты)

10.1 `admin/analytics`: SQL-агрегаты поверх `query_log` - запросы и токены за
     период, доля отказов, доля помеченных «неверными», активность по часам/дням
     недели, топ вопросов.
10.2 Экран: простые таблицы + минимальные графики. Разрез: коллекция
     (collection-admin - свои) и портал (super-admin, плюс срез по пользователям).
- Приёмка: цифры сходятся с содержимым лога на тестовой выборке; collection-admin
  видит аналитику только своих коллекций.

### Этап 11. UI и навигация (собирается параллельно)

11.1 Каркас с сессией: шапка (логин/логаут), меню по роли (вопрос, история; для
     админов - документы, лог, аналитика; для super-admin - пользователи,
     коллекции, калибровка).
11.2 Экран вопрос-ответ под сессией: селектор доступных коллекций, блок истории,
     кнопка обратной связи.
- Приёмка: пункты меню соответствуют роли; чужие экраны недоступны и по прямой
  ссылке (проверять серверные 403, не только скрытие в меню); сквозной сценарий
  залогиненного пользователя работает.

### Этап 12. Упаковка и документация

12.1 `.env.example` - новые ключи (раздел 2). README - bootstrap super-admin
     (`SUPERADMIN_*`, `SECRET_KEY`), загрузка инструкций через UI (CLI-ingest
     остаётся для массовой заливки).
12.2 Compose/Docker: воркер индексации живёт в том же процессе приложения (проверить,
     что фоновый исполнитель стартует в контейнере); миграции на старте.
12.3 Документация (блок M бэклога): `overview.md` (авторизация, логирование с полным
     текстом, аналитика как реализованная), `open-questions.md` (перенос решений в
     принятые, открытыми - LDAP/AD и приватность LLM), `CLAUDE.md`, `maintenance.md`
     (калибровка из UI, добавление коллекции из UI, заведение пользователей).
- Приёмка: на чистой машине `docker compose up` поднимает БД, портал и воркера;
  первый старт создаёт super-admin; загрузка PDF и вопрос-ответ работают сквозняком.

## 6. Тестирование (что покрываем)

- `security`: round-trip хеша пароля; неверный пароль не проходит.
- `auth`/RBAC: аноним -> 401; чужая роль -> 403; `require_collection_access`
  (super_admin проходит, `user` без гранта - нет).
- Изоляция доступа: `user` с доступом к `erp` не получает `zup` в `/collections` и
  в `/ask` (403). Отдельно от изоляции **поиска** (MVP0) - это про права, не про секции.
- Логирование: строка `query_log` пишется на ответ и на отказ; сбой записи не
  роняет ответ (мок insert с исключением).
- История: `/history/{id}` чужой записи - 403/404; пагинация.
- Soft-delete: чанки архивного документа не попадают в выдачу; возврат восстанавливает.
- Индексация: `index_job` проходит статусы; ошибка парсинга -> `error`.
- Существующие тесты MVP0 (поиск, ответ, стрим, изоляция коллекций) остаются
  зелёными - правятся вместе с изменением сигнатур на этапе 1.

## 7. Наблюдаемость

- Прежний структурный лог на запрос сохраняется (MVP0, раздел 7): коллекция,
  кандидаты, лучшее сходство, вердикт, модель, токены, время.
- Теперь то же плюс привязка к пользователю - копится в `query_log` (БД), а не только
  в текстовом логе. Это и есть основа истории и аналитики.
- Воркер индексации логирует переходы статусов `index_job` (queued -> running ->
  done/error) с id задачи и коллекцией.

## 8. Definition of Done (MVP1)

- Портал закрыт авторизацией; бутстрап super-admin создаётся при первом старте.
- Три роли работают: `user` спрашивает по доступным коллекциям; `collection_admin`
  грузит/архивирует инструкции, запускает индексацию, смотрит лог и аналитику своих
  коллекций; `super_admin` ведёт пользователей и коллекции, калибрует пороги,
  смотрит лог и аналитику по порталу.
- Каждый вопрос-ответ (и отказ) пишется в `query_log` с привязкой к пользователю и
  коллекции, полным текстом, токенами и диагностикой.
- История: последние 3 на главной + полная с пагинацией и провалом в ответ.
- Коллекции ведутся в БД; порог правится из UI без деплоя.
- Загрузка PDF и пометка неактуальными работают из UI; индексация - фоновая со
  статусом; архивные исключены из поиска.
- Базовая аналитика считается и совпадает с логом.
- README и docs позволяют развернуть и эксплуатировать без устной передачи знаний.

## 9. Риски и на что смотреть

- **Рефактор коллекций в БД (этап 1) трогает горячий путь поиска.** Секции `chunks`
  завязаны на строковый `code`; рассинхрон `collections` <-> секции ломает поиск.
  Митигируем неизменяемостью `code`, сохранением сигнатур `search`/`answer_*` и
  прогоном тестов поиска сразу.
- **Создание коллекции из UI - многошаговая операция.** Строка + DDL секции + папка
  должны быть согласованы; частичный сбой оставит «полу-коллекцию» - нужен откат.
- **Токены в стриме** приходят финальным чанком с пустым `choices` - запись
  `query_log` на `/ask/stream` должна дождаться его.
- **Безопасность внутренних учёток**: argon2, HttpOnly/SameSite cookie, CSRF для
  админ-форм, отказ неактивным. Базовое закрыть, но не переусложнять.
- **Soft-delete и недобор top-k**: фильтр `archived_at` - джойн после обхода индекса;
  на пилоте пренебрежим, при заметном недоборе денормализовать `archived` в `chunks`.
- **Лог как узкое место надёжности**: запись не должна ронять ответ, но и молча
  терять записи нежелательно - сам сбой логирования логируем.
- **Фоновый воркер на on-premise**: рестарт контейнера теряет незавершённую задачу
  (при старте `running` -> `error`); тяжёлый ingest не должен конкурировать с
  ответами (ограничить параллелизм). Вынос в отдельный процесс - пост-MVP.
- **Калибровка** остаётся чувствительным местом (как в MVP0): golden-наборы малы,
  пороги - стартовые; смотреть распределения, не только рекомендованное число.
