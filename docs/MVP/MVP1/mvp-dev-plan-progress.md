### 2026-07-15. Этап 9 - Калибровка порогов (super-admin)
- Статус: сделано.
- Что сделано:
  - `admin/calibration.py`: `load_questions` (golden-набор из `EVAL_QUESTIONS_FILE`,
    дефолт `eval/questions.yaml`), `run_calibration` (та же механика, что

    делит модель эмбеддингов с ответами).
  - **Правило рекомендации формализовано из решений MVP0 (2026-07-14)**:
    `порог = min(relevant) - 0.03, вниз до сотых`. Негативы выше порога рекомендацию
    не двигают (околодоменные проходят сознательно - закрывает промпт), но
    фиксируются в обосновании; recall@1 по covered < 100% даёт предупреждение
    «сначала чинить поиск». Правило воспроизводит оба принятых порога.
  - Эндпоинты (super-admin): `GET /admin/calibration` (экран),
    `POST /api/admin/calibration/{code}` (запуск; 422 - нет набора, 409 - занято,
    503 - модель не загружена), `GET /api/admin/calibration/{code}` (статус/результат).
    Применение порога - существующий `PATCH /api/admin/collections/{id}` (этап 8),
    нового эндпоинта не потребовалось (J2).
  - UI `templates/admin/calibration.html`: выбор коллекции, запуск с опросом статуса,
    отчёт (рекомендация + обоснование, recall, распределения, таблица вопросов,
    промахи подсвечены), кнопки «Применить рекомендованный» / «Применить свой».
    Пункт «Калибровка» в меню super-admin на всех страницах.
  - `Dockerfile`: `COPY eval ./eval` - golden-наборы нужны калибровке в контейнере;
    конфиг `EVAL_QUESTIONS_FILE` в `.env.example`.
- Проверка приёмки:
  - Тесты (`test_calibration.py`, 11 шт., всего 103): правило воспроизводит числа
    калибровок MVP0 (erp 0.6155/негативы до 0.6094 -> 0.58; zup 0.5857/0.6324 ->
