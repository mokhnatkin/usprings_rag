### 2026-07-15. Этап 2 - Аутентификация
- Статус: сделано.
- Что сделано:
  - Модель `User` (`models.py`) + enum `Role` (user / collection_admin / super_admin);
    миграция 0005 (таблица users, без сида - учётка создаётся bootstrap).
  - `security.py`: `hash_password`/`verify_password` на argon2 (`pwdlib[argon2]`).
  - `auth.py`: провайдер `authenticate` (внутренние учётки - сюда позже ляжет
    LDAP/AD, не трогая эндпоинты), сессия (`login_user`/`logout_user`,
    `current_user_or_none`, `get_current_user` - зависимость 401), `bootstrap_super_admin`.
  - `api.py`: `SessionMiddleware` (SECRET_KEY, HttpOnly, SameSite=Lax; пустой ключ -
    эфемерный dev-режим с предупреждением); `GET/POST /login`, `POST /logout`,
    форма `templates/login.html`; bootstrap в lifespan; `/`, `/ask`, `/ask/stream`,
    `/collections` закрыты авторизацией (аноним на `/` -> редирект на `/login`).
  - Зависимости: `pwdlib[argon2]`, `itsdangerous` (сессии), `python-multipart` (формы).
  - Конфиг: `SECRET_KEY`, `SESSION_COOKIE_NAME`, `SESSION_MAX_AGE`, `SUPERADMIN_LOGIN`,
    `SUPERADMIN_PASSWORD` в `.env`/`.env.example`.
- Проверка приёмки:
  - Bootstrap идемпотентен: пустая таблица -> 1 super-admin; повторный вызов -> без
    дублей (проверено вживую: users 0 -> 1 -> 1; auth верный/неверный пароль).
  - Тесты (`test_security.py`, `test_auth.py`): аноним на `/` -> 303 `/login`;
    `/ask` и `/ask/stream` без сессии -> 401; вход верными кредами -> сессия и `/`
    открыт; логаут рвёт сессию; неверный пароль и неактивная учётка -> `/login?error`.
    TestClient без lifespan (BGE-m3 не грузится). Всего 36 тестов зелёных.
- Заметки:
  - `same_site=lax`, `https_only=false` - on-premise может работать по http; за TLS
    включить `https_only=True`. CSRF для админ-форм - на этапах с изменяющими
    действиями (3+).
  - Смена своего пароля (B6) и RBAC/доступ к коллекциям - этап 3.

### 2026-07-15. Этап 1 - Коллекции в БД (фундамент)
- Статус: сделано.
- Что сделано:
  - Модель `CollectionRow` + миграция 0004 (таблица `collections`), сид `erp` (0.58,
    its_erp) и `zup` (0.55, its_zup) из прежнего enum.
  - `collection.py` -> тонкая read-model поверх БД: `get_collection`/`list_collections`
    читают справочник (кэш в памяти, `invalidate_cache`); объект `Collection` тот же
    (`code`/`title`/`folder`/`threshold`), поэтому `retrieval.search`/`answer_*` не
    менялись. `CollectionCode`/`DEFAULT_COLLECTION` остались как константы для
    сида/дефолта, не как источник истины.
  - Потребители переведены на БД: `api.py` (`/collections`, `_resolve_collection` ->
    422 на неизвестной), `ingest/cli.py`, `search_cli.py`, `eval/run_eval.py` (убраны
    статические `choices`, валидация через `get_collection`).
  - `collections_service.create_collection`: строка + `CREATE TABLE ... PARTITION OF
    chunks` + папка, в одной транзакции (DDL Postgres транзакционен), с проверкой
    формата кода; сбрасывает кэш.
  - `alembic/env.py`: `include_object` игнорирует дочерние секции `chunks_<code>` -
    иначе autogenerate на каждую секцию предлагал бы drop_table (ложный дифф).
- Проверка приёмки:
  - `alembic check` - "No new upgrade operations detected" (модель = схема).
  - Тесты изоляции коллекций и логики поиска зелёные без изменения сигнатур `search`;
    новый `test_collections_service.py`: read-model видит сид, неизвестный код -
    ValueError, `create_collection` заводит строку + секцию (в плане запроса
    `chunks_<code>`) + папку. Всего 27 тестов.
  - `grep`: бизнес-логики на enum как источнике истины не осталось (только
    `DEFAULT_COLLECTION` и конструирование `Collection` в логических тестах).
- Заметки:
  - `code` коллекции неизменяем (имя секции `chunks_<code>`); правка кода запрещена,
    меняем title/threshold/is_active (админка - этап 8).
  - Логические тесты (`test_stream`, `test_collection_isolation`) конструируют
    `Collection` напрямую - не зависят от БД-справочника при импорте.

  1. Аутентификация - **только внутренние учётки** (логин+пароль, cookie-сессии,

     «неверных», активность, топ вопросов).

     списков/аналитики).
  6. Пользователь **сам меняет свой пароль**; отметка «ответ неверный» - с
     опциональным комментарием.
- **Порядок работ:** этап 1 (коллекции в БД) - первый и фундаментный, трогает
  горячий путь поиска; далее auth -> доступ -> логирование -> история -> админка ->
  аналитика -> упаковка.

  2026-07-15. Открытыми на проекте остаются LDAP/AD и приватность LLM
