# Обзор проекта и архитектура пилота

## Цель

Снизить время поиска ответов в инструкциях для пользователей ИТ-процессов (пилот - 1С ERP):
дать веб-портал с ИИ-ассистентом, который отвечает на вопросы «как сделать ту или иную
операцию», опираясь на загруженные инструкции.

## Пользователи и охват

- Пилот: пользователи 1С ERP (ИТ-процессы).
- Далее: документация других служб и процессов - другие ИТ-системы, техническая
  дирекция и др. Архитектура должна допускать несколько наборов знаний (по подразделениям).

## Основные сценарии

1. Пользователь задаёт вопрос на портале.
2. Система векторизует вопрос и ищет релевантные фрагменты инструкций в pgvector.
3. Если релевантность выше порога - LLM (через OpenRouter) формирует ответ с опорой
   на найденные фрагменты и ссылками на источник.
4. Если релевантного документа нет - вежливый отказ (реализация «белого списка вопросов»).
5. Администратор загружает новые инструкции, они обрабатываются и попадают в базу знаний.

## Компоненты (целевые для пилота)

- **Портал / API** - FastAPI: приём вопроса, оркестрация поиска и генерации ответа.
- **Векторное хранилище** - PostgreSQL + pgvector: чанки инструкций и их эмбеддинги.
- **Пайплайн загрузки** - парсинг документов (Word/PDF), разбиение на чанки,
  расчёт эмбеддингов, запись в базу.
- **Генерация ответа** - вызов LLM через OpenRouter с контекстом из найденных чанков.
- **Админский модуль** - загрузка инструкций и управление базой знаний.
- **Фронтенд** - простой (серверные шаблоны / HTML+JS), нарастить до React при необходимости.

## Технологические решения

| Область | Выбор |
|-|-|
| Бэкенд | Python + FastAPI |
| Векторное хранилище | PostgreSQL + pgvector |
| Фронтенд | Простой (серверные шаблоны / HTML+JS) |
| LLM | OpenRouter |
| Эмбеддинги | Не выбрано (см. open-questions) |
| Инфраструктура | Docker Compose, on-premise |

Стек согласован с целевой архитектурой группы (ИТ-стратегия, PostgreSQL/Python/React/Docker).
Для пилота фронтенд намеренно упрощён.

## Белый список разрешённых вопросов

Реализуется семантически, а не ручным перечнем: ответ выдаётся только если найден
документ с оценкой сходства выше порога. Порог подбирается на этапе прототипа.
Плюс - гибкость и отсутствие ручного ведения списка; риск - нужно откалибровать порог,
чтобы не отвечать на нерелевантное и не отказывать на релевантное.

## Этапы (черновой план)

1. Сбор набора частых инструкций 1С ERP и приведение к тексту.
2. Прототип пайплайна: парсинг -> чанкинг -> эмбеддинги -> pgvector.
3. Прототип ответа: поиск + генерация через OpenRouter, калибровка порога.
4. Портал (простой UI) поверх API.
5. Админский модуль загрузки инструкций.
6. Пилотная эксплуатация, сбор обратной связи, расширение на другие службы.
