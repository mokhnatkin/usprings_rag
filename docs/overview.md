# Обзор проекта и архитектура пилота

## Цель

Снизить время поиска ответов в инструкциях для пользователей ИТ-процессов (пилот - 1С ERP):
дать веб-портал с ИИ-ассистентом, который отвечает на вопросы «как сделать ту или иную
операцию», опираясь на загруженные инструкции.

## Пользователи и охват

- Пилот: пользователи 1С ERP (ИТ-процессы). Система используется как база знаний
  по использованию ИТ-систем.
- Далее: документация других служб и процессов - другие ИТ-системы, техническая
  дирекция и др. Архитектура изначально должна закладывать масштабирование на другие
  подразделения и процессы: несколько независимых наборов знаний с разграничением
  по подразделению/процессу.

## Основные сценарии

1. Пользователь задаёт вопрос на портале.
2. Система векторизует вопрос и ищет релевантные фрагменты инструкций в pgvector.
3. Если релевантность выше порога - LLM (через OpenRouter) формирует ответ с опорой
   на найденные фрагменты и ссылками на источник.
4. Если релевантного документа нет - вежливый отказ (реализация «белого списка вопросов»).
5. Администратор загружает новые инструкции, они обрабатываются и попадают в базу знаний.

## Компоненты (целевые для пилота)

- **Портал / API** - FastAPI: приём вопроса, оркестрация поиска и генерации ответа.
- **Векторное хранилище** - PostgreSQL + pgvector: чанки инструкций и их эмбеддинги.
- **Пайплайн загрузки** - парсинг документов (Word/PDF; markdown из выгрузки
  1С:ИТС - пост-MVP), разбиение на чанки, расчёт эмбеддингов, запись в базу.
- **Генерация ответа** - вызов LLM через OpenRouter с контекстом из найденных чанков.
- **Админский модуль** - загрузка инструкций и управление базой знаний.
- **Фронтенд** - простой (серверные шаблоны / HTML+JS), нарастить до React при необходимости.

## Технологические решения

| Область | Выбор |
|-|-|
| Бэкенд | Python + FastAPI |
| Векторное хранилище | PostgreSQL + pgvector |
| Фронтенд | Простой (серверные шаблоны / HTML+JS) |
| LLM | OpenRouter |
| Эмбеддинги | Локальная BGE-m3 (multilingual, dim 1024), on-premise |
| Аутентификация | Внутренние учётки (argon2, cookie-сессии); LDAP/AD - пост-MVP |
| Инфраструктура | Docker Compose, on-premise |

Стек согласован с целевой архитектурой группы (ИТ-стратегия, PostgreSQL/Python/React/Docker).
Для пилота фронтенд намеренно упрощён.

## Авторизация и роли (реализовано в MVP1)

- **MVP0**: авторизация не требовалась - доступ открытый.
- **MVP1**: портал закрыт входом. Аутентификация - **внутренние учётки** (логин +
  пароль, хеш argon2), cookie-сессии (Starlette `SessionMiddleware`, `SECRET_KEY`).
  При первом старте с пустой таблицей `users` бутстрапится один **super-admin** из
  `SUPERADMIN_LOGIN`/`SUPERADMIN_PASSWORD`. Провайдер аутентификации спрятан за
  тонким интерфейсом - LDAP/Microsoft AD добавляется как вторая реализация без
  переработки потока (пост-MVP).
- **Три роли** (одна на пользователя): `user` (спрашивает по доступным коллекциям),
  `collection_admin` (по своим коллекциям: загрузка/архивация инструкций, журнал,
  аналитика), `super_admin` (пользователи, коллекции, калибровка порогов, журнал и
  аналитика по всему порталу). Доступ к коллекциям - таблица
  `user_collection_access`; super-admin грантов не требует. Разграничение держат
  серверные зависимости (401/403), а не только скрытие пунктов меню.

## Логирование запросов и аналитика (реализовано в MVP1)

Каждый вопрос-ответ (и отказ) пишется в таблицу `query_log` в привязке к
пользователю и коллекции. По каждому запросу:

- timestamp, ID пользователя, ID коллекции;
- **полный текст вопроса и полный текст ответа** (у отказа - текст отказа);
- количество потраченных токенов (prompt/completion/total; у отказа - 0, LLM не звали);
- исход запроса: `answered` или `refused` (нет релевантного документа выше порога);
- время ответа (в секундах);
- лучшее сходство (по нему видно поведение порога на реальных вопросах);
- ID модели LLM и список найденных документов (`sources`) - диагностика для разбора жалоб;
- оценка пользователя (см. ниже) - если он её поставил.

Текст храним **полностью** (нужен истории пользователя и переиспользованию жалоб в
eval). `QUERY_LOG_PREVIEW_CHARS` (`.env`) - только усечение вопроса/ответа в списках
и аналитике, не для хранения. Запись лога обёрнута так, чтобы её сбой **не ронял**
ответ пользователю. Доступ к журналу: super-admin - весь портал, collection-admin -
свои коллекции (`/admin/logs`).

### Обратная связь по ответу (реализовано в MVP1)

После получения ответа пользователь может пометить его как **неправильный или
некорректный** (кнопка в портале). Отметка сохраняется в тот же лог **вместе с вопросом
и ответом** - иначе жалоба бесполезна: без пары «вопрос-ответ» непонятно, что чинить.

Зачем это нужно именно нам: качество RAG-ответа ломается в трёх разных местах, и по
жалобе с сохранённым контекстом видно, в каком именно:

- **поиск** - нашлись не те фрагменты (лечится чанкингом, `TOP_K`, реранкером);
- **порог** - вопрос отсекло, хотя инструкция есть, либо пропустило непокрытый
  (лечится калибровкой порога коллекции, см. `maintenance.md`);
- **генерация** - фрагменты верные, но модель ответила плохо (лечится промптом или
  сменой модели через A/B).

Поэтому в записи о жалобе полезно хранить и диагностику: лучшее сходство, список
найденных документов, ID модели. Разметка «неправильно» на реальных вопросах - это ещё
и **готовый материал для eval-набора**: жалобы пополняют `eval/questions.yaml`, а
исправление проверяется прогоном (см. `maintenance.md`).

Реализовано минимально: одна кнопка «ответ неверный» с опциональным полем
комментария (`feedback_comment`). Пометить можно только свою запись. Развёрнутая
шкала оценок не нужна: цель - поймать проблемные ответы, а не измерять
удовлетворённость.

### Аналитика (реализовано в MVP1)

Экран `/admin/analytics` считает базовые агрегаты поверх `query_log`:

- количество запросов и токенов за период;
- активность по дням недели и часам;
- доля отказов (запросы без релевантного документа) - сигнал о пробелах в базе знаний;
- **доля и топ ответов, помеченных как неправильные** - главный сигнал качества:
  показывает, какие инструкции надо дописать, а где чинить поиск, порог или промпт;
- топ вопросов (с числом отказов) и срез по пользователям (только super-admin).

Разрез: collection-admin - свои коллекции, super-admin - весь портал плюс срез по
пользователям. Графики - на CSS без внешних библиотек (on-premise, без CDN).
Развёрнутый дашборд (фильтры, drill-down, экспорт) - пост-MVP.

## Белый список разрешённых вопросов

Реализуется семантически, а не ручным перечнем: ответ выдаётся только если найден
документ с оценкой сходства выше порога. Плюс - гибкость и отсутствие ручного ведения
списка.

Порог - **свойство коллекции**, а не одна константа на систему: у каждой базы знаний
свой корпус и своя лексика, поэтому и распределения сходств разные. Прежнее значение
(`0.53`, recall@1 = 17/17) откалибровано на тестовом корпусе из 8 инструкций, который
удалён (решение от 2026-07-14): **оно больше не действующее**, пороги калибруются
заново и отдельно для `erp` и `zup` (см. `MVP/MVP0/backlog.md`).

Работает порог в паре с промптом - это две линии защиты, и у связки есть известный предел:
порог надёжно режет вне-доменные вопросы, но околодоменные (про 1С, но не покрытые
нашими инструкциями) проходят его и отсекаются уже промптом («в инструкциях этого
нет»). Если же в базе есть лексически похожий материал, модель ответит по нему - не
выдумывая, но и не по существу вопроса. Механика и предел подробно - в
`how-it-works.md`; обоснование порога и смягчение (реранкер, пост-MVP) -
в `open-questions.md`.

## Исходные инструкции

Корпус базы знаний - документация 1С:ИТС (its.1c.ru, есть подписка), выгруженная
парсингом (см. `docs/1C_parsing/parsing-guide.md`). Инструкции разложены по
**коллекциям**, папка = коллекция:

| Коллекция | База ИТС | Папка | Объём |
|-|-|-|-|
| `erp` (1С:ERP) | `erp25ltsdoc` | `docs/manuals/its_erp/` | 411 документов, ~800 МБ (реестр 3645 строк) |
| `zup` (1С:ЗУП) | `zupcorpdoc` | `docs/manuals/its_zup/` | 194 документа, ~12 МБ (реестр 194 строки) |

Единица выгрузки - глава: подразделы портала обычно якоря внутри одной главы,
поэтому у ERP строк реестра примерно в 9 раз больше, чем документов (у ЗУП
оглавление плоское - строка = документ).

Парсер отдаёт PDF с картинками (текст + скриншоты, сборка pandoc+typst) - этот
вариант индексируется ingest как есть (`uv run ingest --collection erp`). Markdown-
вариант (только текст) - после добавления md-парсера (пост-MVP).

Стартовый тестовый набор из 8 инструкций (`docs/manuals/IT_1C/`) удалён: на нём
отрабатывалась технология, в базе знаний он не нужен (решение от 2026-07-14).

## Этапы (черновой план)

1. Сбор набора частых инструкций 1С ERP и приведение к тексту.
2. Прототип пайплайна: парсинг -> чанкинг -> эмбеддинги -> pgvector.
3. Прототип ответа: поиск + генерация через OpenRouter, калибровка порога.
4. Портал (простой UI) поверх API.
5. Админский модуль загрузки инструкций.
6. Пилотная эксплуатация, сбор обратной связи, расширение на другие службы.
